## On the project

The gpo-lens codebase is in a good shape. The architecture boundary (core never imports web/narration) is clean and enforced by AST-based tests. The existing pattern of pure scanner functions in `detection.py` → typed dataclasses → `estate_doctor()` aggregation → `DoctorFinding` is well-established and makes adding new detectors straightforward. The `danger.py` module follows this pattern exactly, which is the right call.

The main tension I see is between the "air-gappable, stdlib-only core" and the web UI's real-world usability demands. The web UI is where this tool will actually be used day-to-day, and the Plans 017/018 changes make it meaningfully more useful (searchable directory, human-readable policy names, danger detectors). But the danger detectors are only as good as their calibration — without real-estate validation (WI-032), they're well-structured but unproven.

## On the work done

**What went well:**
- The subagent delegation pattern worked well for the initial implementation. GLM implemented Plan 017 cleanly, Kimi implemented Plan 018 Phase A + the MS16-072 fix.
- The adversarial reviews were genuinely valuable — the first reviewer caught that the Plan 017 tests were completely missing (the subagent claimed they existed but they didn't), and the second reviewer caught the `overbroad_apply_group_policy` SDDL fallback gap and the Owner SID check gap.
- The WI-028 fix was satisfying — the root cause was clear (the parser only handled `Security` CSE's `SettingString` children, not the real-world `Policy > DropDownList > Value > Name` structure), and the calibration test against the real work estate confirmed all 28 loopback GPOs now resolve to merge/replace.

**What I'm less confident in:**
- The `danger_rules.toml` registry paths were corrected by the subagent from the plan's draft paths to "technically accurate Windows registry locations." I verified the paths look right (WDigest is under `SYSTEM\CurrentControlSet\Control\Lsa`, not `Software\Policies\...`), but without a real estate that has these settings configured, I can't confirm the identity matching works end-to-end against real GPO data. This is WI-032.
- The MS16-072 fix changed calibration numbers (work domain went from 112 to 24 vulnerable GPOs). The fix is correct per Microsoft's GPMC documentation (Apply Group Policy = Read+Apply), but the "edit" prefix matching is still slightly loose — it could match "Edit settings and then custom" which is a GPMC option that doesn't necessarily include Read. I tightened it to `startswith("edit ")` or `startswith("edit,")` but there may be other locale-specific variants.
- The `_setting_label` helper is called unconditionally in some templates even when `admx is None`, which is a minor perf waste (no functional issue). The adversarial reviewer flagged this but it's not worth fixing right now.

**The MS16-072 fix is the highest-risk change.** It changes the semantics of a security-relevant detector. The old code produced false positives on every default-filtered GPO (treating "Apply Group Policy" as non-Read); the new code is correct per Microsoft's documentation but the calibration numbers shifted dramatically (112→24). If the Microsoft documentation is wrong or I'm misreading it, this is a regression. The calibration test against the real estate at least confirms the new numbers are stable.

## On what remains

**Needed before this can ship to a production audience:**
1. **WI-032** — Calibration tests for the danger rules against a real estate. Without these, the danger detectors are well-structured but unproven. The registry paths in `danger_rules.toml` could be wrong and nobody would know.
2. **WI-031** — `danger_findings()` memoization. On a 129-GPO estate, the dashboard renders 3x the danger evaluation work. This is a real perf issue at scale.
3. **WI-033** — `q` length limit. Low severity but trivial to fix.

**Nice to have:**
- The `_setting_label` helper should be memoized or only called inside `{% if admx %}` blocks.
- The `/danger` route lacks a `sort` parameter (unlike `/` and `/ou`).
- The `gpo_writable_by_nonadmin` detector doesn't exclude built-in operator groups (Account Operators S-1-5-32-548, Server Operators S-1-5-32-549, Print Operators S-1-5-32-550) from the default-writer set. These are pre-existing limitations in `_is_default_writer_sid` that Plan 018 inherited.
- Plan 019 (scope resultant and gate attribution) depends on WI-028, which is now resolved. It could be the next implementation target.

## Gaps to flag

- **`danger_rules.toml` registry paths unverified against real GPO data.** The subagent corrected them from the plan's draft, but without a real estate that has WDigest/SMB-signing/LM-hash settings configured, the identity matching is untested against real-world data. (`src/gpo_lens/danger_rules.toml`, WI-032)
- **MS16-072 `edit` prefix matching may be too loose for locale variants.** The fix uses `startswith("edit ")` or `startswith("edit,")`, but GPMC in non-English locales may produce different permission labels. No test exercises a non-English permission string. (`src/gpo_lens/detection.py:297`)
- **`estate_summary()` doesn't pass `admx` to `danger_findings()`.** If a user supplies name-keyed danger rules via `GPO_LENS_DANGER_RULES_DIR`, the dashboard posture count will differ from the `/danger` view count. The shipped rules are all identity-keyed so this is theoretical, but it's a silent divergence. (`src/gpo_lens/queries.py:472`, WI-031)
- **`overbroad_apply_group_policy` SDDL fallback uses `WD`/`AN` SDDL abbreviations.** Real SDDL strings use full SIDs (`S-1-1-0`), but the SDDL parser returns abbreviations as-is. The detector now handles both forms, but this is a parser-level inconsistency that could bite elsewhere. (`src/gpo_lens/danger.py:91`)
- **No calibration test for Bucket 2 structural detectors.** `gpo_writable_by_nonadmin`, `local_admin_push`, and `overbroad_apply_group_policy` have unit tests with synthetic data but no real-estate validation. The fixture estate's `gpo-cpassword` GPO triggers `local_admin_push` (1 finding), but the SDDL-based detectors are untested against real GPO ACLs.
