## On the project

The architecture continues to be the project's strongest asset. The deterministic/AI separation enforced by `tests/_arch.py` is real, tested, and survived every change in this session — including the B5 change that pulled `query_dispatch`'s `__import__` into a static import (which actually *strengthened* the boundary: the AST checker now sees the merge dependency that was previously invisible). The `query_dispatch` table pattern continues to pay off; the new `danger`/`resultant`/`trends` contract tests were nearly free because the dispatch design made the surface enumerable.

What's most striking after this session is how much the latest reflection (`2026-06-25-glm-5-2-2`) got *wrong* about project state. It claimed "no GitHub remote, no CI, no tags, no CHANGELOG" — but all of those landed by v0.7.x. The Plan 010 framing of "Phase 0 still most important" was outdated. I corrected this in the plan and in the gaps section of this reflection, but it's a useful reminder that reflections themselves drift and need reconciliation just like work items do. Future sessions should run `git remote -v` and `git tag` before quoting a prior reflection's infra claims.

The thing I'm least confident in after this session is the **test coverage gate at 85% reporting 20.7%** when sample tests skip. That's a CI false-confidence hazard that the latest reflection didn't flag. The gate can pass locally with `samples/` present and fail on a fresh checkout, or vice versa. Worth a WI of its own (not filed this session — should be).

## On the work done

The implementer+adversarial-reviewer pattern with different model lineages worked exactly as the prior reflection described. The Kimi reviewer caught three things I would have shipped:

1. `ruff` lint failures in the new tests (line length, unused var, import sort) — I had run ruff on `src/` but not the new test files.
2. `UnicodeDecodeError` from `ET.parse` on encoding-corrupt Preferences XML would have crashed `load_estate` — exactly the input class the new scanner was meant to surface as a coverage gap.
3. The `dest.iterdir()` call in `_safe_extract` cleanup was outside the inner try/except, so a missing `dest` would mask the original extraction error.

Each of these is exactly the kind of thing a second pair of eyes with a different reasoning style catches. The cost (~2x implementation time) is worth it. I'd recommend this as the default pattern for any non-trivial fix session.

The trickiest fix was A3 (`parse_report` + `parse_report_xml`). The original code had a comment claiming "the report may contain one `<GPO>` as the root itself" but the implementation skipped the root via `gpo_elem is root` — a direct contradiction. My first attempt at a fix broke the existing tests because both wrapper and single-GPO shapes use `<GPO>` as the root localname; the distinguishing signal is the presence of a *nested* `<GPO>` child. The reviewer then pointed out that an empty wrapper would still be mis-parsed, so I added a `has_content` clause (require at least one non-`<GPO>` child for the single-GPO interpretation). This is the kind of multi-step refinement that benefits from a reviewer who actually traces through the edge cases.

What I'm least confident in: the **`load_danger_rules` RuntimeError**. The fail-fast policy for a zero-rule shipped file is the right call for a security tool, but it's a behavior change that could surprise an operator who legitimately wants to run with override-only rules. I documented it in `wi_danger.md` AC-13, but a deployment that's been silently relying on the old "return [] and continue" behavior would now crash on the next ingest. Worth calling out in the CHANGELOG for the next release.

## Gaps to flag

- **The test coverage gate (85%) reports 20.7% on a fresh checkout** because sample-dependent calibration tests skip when `samples/` is absent. This is a false-confidence hazard: CI can pass locally while a fresh checkout fails the gate, or vice versa. Either lower the threshold to reality or build the synthetic fixture estate Plan 010 WI-0.2 calls out. Not filed as a WI this session — should be.

- **`load_danger_rules` RuntimeError is a behavior change that could surprise existing deployments.** A site that has been silently running with a corrupt or empty shipped `danger_rules.toml` (silently, per the old behavior) will now crash on the next ingest. The fail-fast policy is correct, but the CHANGELOG should call this out for the next release so operators aren't blindsided. `src/gpo_lens/danger.py:555`.

- **The `dest.iterdir()` guard in `_safe_extract` cleanup is correct but untested.** I added the `dest.is_dir()` check per the reviewer's feedback, but there's no test that exercises the "cleanup runs when dest is missing" path. The existing `_safe_extract` tests cover zip-slip and size limits; the missing-dest cleanup path is a regression-risk gap. `src/gpo_lens/web/app.py:121-138`.

- **`_scan_corrupt_gpp_xml` and `_scan_sysvol_coverage` double-walk Preferences.** Filed as WI-066 but worth restating: on a ~100-GPO estate with several thousand Preferences files, this is a measurable per-ingest regression. The two scanners should share one directory walk. `src/gpo_lens/ingest.py:1299,1343`.

- **The latest reflection's project-state claims were outdated.** `reflection-2026-06-25-glm-5-2-2` said "no GitHub remote, no CI, no tags, no CHANGELOG" — all of those landed by v0.7.x. I corrected Plan 010's framing but the older reflection file itself is still on disk and will mislead any agent that reads it without checking git state. Consider an addendum or strikethrough on prior reflections when they're contradicted by later work, or a convention that reflections are point-in-time and should be reconciled against git at session start (which the `/start` skill already does for WIs but not for reflection claims).

- **No WI links in active breadcrumbs.** Filed as WI-067, restating here for visibility: the 5 active breadcrumbs are floating enhancement notes with no numbered-WI backing. Their status (in-flight? backlog? abandoned?) is ambiguous from the filesystem alone.

- **Compliance control IDs in `danger_rules.toml` remain unverified against authoritative benchmarks.** The prior reflection flagged this; it's still true. The CIS/STIG/NIST mappings are plausible but not cross-referenced. Reviewer with benchmark access needed before public publication.
