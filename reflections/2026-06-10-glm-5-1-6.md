## On the project

Plan 010 Workstreams A/B/C were all already done before this session — the changelog, delegation deep-dive, SDDL parsing, estate reports, narration layer, NL query routing. The codebase has strong conventions (no comments, stdlib-only core, spec-driven ACs, canonical GUID everywhere) and the test suite is thorough (389 tests, ruff+mypy--strict clean).

The main structural issue is `queries.py` at 1,546 lines — Plan 010 says "don't decompose as a standalone session," which is wise, but it's the next big file to split when a Workstream B item opens it.

## On the work done

**Player-coach pattern worked well.** Dispatching GLM and Kimi in parallel for independent tasks, then both adversarial reviewers in parallel, kept the session moving. The adversarial reviews were high-value:

- Phase 0: Both reviewers caught the same 2 SHA mismatches (setup-python and setup-uv pinned to post-release commits, not tagged releases). This was the single most valuable finding — it defeated the purpose of SHA pinning.
- Workstream D: Reviewer 1 caught the architecture guard test regression (C1 — the test was only scanning `__init__.py` after decomposition, not the full package). Reviewer 2 caught the `_get_estate()` `sys.exit()` inconsistency (C2). Both important, both fixed.

The GLM agent's CLI decomposition was clean — 13 modules, correct import graph, no circular imports. The Kimi agent's collector hardening added DryRun, progress, validation without breaking the original flow.

**What I'd want checked again:** The `_get_estate()` change from `sys.exit(2)` to `raise FileNotFoundError`. This is correct — `main()` catches `FileNotFoundError` and returns 2 — but it changes the error path for any code that calls `_get_estate()` directly. I verified that only CLI cmd_* functions call it, but a second pair of eyes is always good on error-path changes.

## On what remains

Plan 010 is now **fully complete** — Phase 0 + Workstreams A/B/C/D. The repo is public, CI is green, v0.2.0 is tagged.

Beyond the plan, the remaining opportunities are:

1. **queries.py decomposition** — 1,546 lines, flagged in Plan 010 as "gate it behind a Workstream B item." The `detection.py` extraction was already done. The remaining groups (topology, baseline, settings) could be split when next touched.
2. **`_VALID_QUERIES` / `_QUERY_DISPATCH` drift** — Two separate data structures that must stay in sync. A union or assertion would be safer.
3. **CLI test coverage gaps** — No CLI-level tests for `report`, `changelog`, or `baseline-diff` subcommands (the underlying query functions are tested, but the CLI wiring isn't).
4. **PowerShell edge cases** — Empty domain (0 GPOs), GPO names with reserved device names (CON, PRN), trailing dots/spaces in display names.
5. **`getattr(args, "side", None)` pattern** — Could use argparse `parents=` to share filter arguments instead of defensive getattr.
6. **Performance testing** — No tests with 500+ GPO estates. The `settings_at_som` call in `report` iterates all SOMs, which could be slow.

## Gaps to flag

- `src/gpo_lens/cli/_report.py:41` — `baseline_diff(estate, baseline_settings)` passes `None` for the ADMX param. This works because `baseline_diff` defaults to an empty `PolicyDefinitions()`, but it means baseline-diff from the report command never uses ADMX crosswalk even if `--admx-dir` is available on other commands. The `report` subcommand doesn't have an `--admx-dir` flag.
- `src/gpo_lens/cli/_narration.py:49-87` — `_QUERY_DISPATCH` dict-of-lambdas is separate from `narration.py`'s `_VALID_QUERIES` set. Adding a query to one but not the other is a silent bug.
- `src/gpo_lens/cli/_helpers.py:17` — `_get_estate` no longer has `sys` import (removed during cleanup). If anyone adds `sys.exit()` back, they'll get an `F821`.
- `scripts/Export-GpoEstate.ps1:57-63` — The writability test runs before DryRun check, creating a test file on disk even in dry-run mode.
- The `.claude/` directory was untracked but a `.claude/settings.json` was previously committed — now removed from tracking and gitignored. Future clones won't have it.
