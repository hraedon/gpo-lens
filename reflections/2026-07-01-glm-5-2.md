## On the project

The codebase is well-structured for its purpose — read-only GPO analysis
with a deterministic core and optional LLM narration. The import boundary
discipline (core must never import narration/web) is enforced and
respected. Test coverage is strong (85% gate, 556 tests). The
dataclass-as-contract pattern in model.py works well — types are the
spec, and the spec files (docs/spec/wi_*.md) provide the acceptance
criteria.

What feels fragile: the token construction in merge.py had a real
security gate bypass (forward expansion of group members) that survived
multiple review rounds and was even "tested" by a test that asserted the
wrong behavior as correct. The SDDL fallback path conflates Read with
Apply rights (WI filed but not fixed — regista migration blocking
breadcrumb filing). The snapshot changelog was silently incomplete
(omitting added/removed GPOs) and no one noticed until this review.

The regista infrastructure is currently blocking — migration 37 is
pending and fails due to insufficient DB permissions. This means no
breadcrumbs can be filed until a DBA applies the migration.

## On the work done

**What went well:** Deploying 4 adversarial reviewers in parallel was
efficient and produced comprehensive coverage. The findings were
well-categorized and mostly accurate. The second-round review caught
important residual issues (stale docstring, cleanup path following
symlinks, missing edge-case tests) that the first round missed.

**What I'm confident in:** The build_token fix is correct — removing
forward expansion is the right approach because upward closure via
reverse expansion handles all nested group cases. The
augment_blocked_registry_from_pol fix is straightforward. The
_safe_extract symlink fix is correct (check before resolve). The
restricted_groups_mode fix is a clean set-based approach.

**What I'd want a second pair of eyes on:** The snapshot_changelog
addition of gpo_added/gpo_removed entries changes the output shape —
existing consumers may not expect these new entry kinds. The sort
key `(gpo_id, side)` is a reasonable default but may not match all
caller expectations. The danger.py change from string matching to
permission_implies_apply() is correct for delegation entries but the
SDDL fallback path still uses the coarser READ_OR_APPLY_RIGHTS check,
which is a known unaddressed issue.

## On what remains

- **SDDL Read vs Apply conflation** (medium, filed as breadcrumb but
  regista is blocking): The SDDL fallback in both merge.py and danger.py
  treats Read-only grants as if they confer Apply rights. This causes
  false positives in danger detection and false positives in the
  security filter gate.
- **WAL/SHM first-write permission window** (low): restrict_db_permissions
  runs at init time but WAL files are created lazily on first write.
  They'll have umask permissions until the next connection's init.
- **events.py integrity** (high, from first-round review): No hash chain
  or tamper detection on the append-only event store.
- **Rate limiting on expensive endpoints** (low): /ask and /ingest have
  no request-rate limiting.
- **regista migration 37** (infrastructure): Needs DBA to apply —
  blocking all breadcrumb/memory operations for this project.

## Gaps to flag

- `merge.py:541-547` — the forward expansion removal is correct but the
  caveat logic at lines 548-553 still fires for unresolved group SIDs
  that were in the token via reverse expansion. The caveat message says
  "unresolved group SID" which is accurate but could be confused with
  the old forward-expansion behavior.
- `danger.py:323-367` — SDDL fallback uses READ_OR_APPLY_RIGHTS which
  includes GR (Read). A GPO with only Read granted to Everyone will be
  flagged as overbroad-apply, which is a false positive.
- `merge.py:635-655` — Same READ_OR_APPLY_RIGHTS issue in the security
  gate's SDDL fallback: a GPO with only Read to a trustee will pass the
  gate (token intersection), allowing the GPO to apply when it shouldn't.
- `snapshot_diff.py:237` — `results.sort(key=lambda e: (e.gpo_id, e.side))`
  sorts all entries by GPO ID then side. The `gpo_added`/`gpo_removed`
  entries have `side=""` which sorts before any real side value. This is
  fine but worth being aware of.
- `store.py:314-326` — WAL/SHM chmod is best-effort and only runs at
  init_db time. Files created by a subsequent write won't be restricted
  until the next connection.
- `web/app.py:130-135` — Cleanup path now checks `child.is_symlink()`
  before `child.is_dir()`, but there's a TOCTOU gap between the two
  checks. Low risk in the web upload scenario.
- Regista migration 37 is blocking all agent-notes operations for
  gpo-lens. This means reflections can be written to disk but not
  ingested into the memory store, and no breadcrumbs can be filed.
