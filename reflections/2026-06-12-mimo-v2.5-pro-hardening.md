# Session Reflection — 2026-06-12

**Work summary:** Implemented the web-UI hardening quartet (symlink-through-zip, streaming uploads, zip-slip/size-cap tests, prompt interpolation) plus seven follow-on fixes surfaced by three rounds of adversarial review. 9 files changed, +419/-92 lines, 15 new security tests.

---

## On the project

The architecture is solid — the separation between deterministic core (tiers 1-2.5) and optional LLM narration is clean and well-enforced by architecture tests. The web UI bolted on top is functional but was written with a "trust the user" mindset that assumed local-only deployment. The security posture was weaker than the rest of the codebase warranted: no symlink checks in zip extraction, full-buffer uploads before size checks, raw user input flowing into LLM prompts. These are all standard web hardening gaps — nothing surprising for a tool that started as a CLI and grew a web face.

The `query_dispatch.py` single-source-of-truth pattern (VALID_QUERIES derived from _QUERY_DISPATCH keys) is elegant. The three-table drift risk (_QUERY_DISPATCH, QUERY_REQUIRED_PARAMS, _PARAM_VALIDATORS) is the kind of thing that will bite someone in six months when they add a new query. The architecture test we added catches it.

## On the work done

The multi-agent approach worked well for this session. GLM and Kimi handled the initial implementation in parallel, and the adversarial reviewers found real issues — the delimiter breakout (C1), the `estate` LLM override (NEW-2), and the routing LLM having zero injection defenses (C2) were all genuine vulnerabilities that the initial implementation missed. Three rounds of adversarial review was the right number; round 3 found only one minor gap.

The prompt injection hardening is the weakest part of the work, not because the code is wrong but because prompt injection is fundamentally unsolvable with string sanitization alone. The delimiter framing + newline stripping + system prompt warning is defense-in-depth, but a sufficiently creative attacker can still craft inputs that confuse the LLM. The 500-char truncation is probably the most effective defense — it limits the attack surface mechanically.

The `_safe_extract` function now has four layers of defense (symlink check, pre-extract path check, post-extract path check, uncompressed size check). The header-based size check is a known limitation (WI-006) — `info.file_size` is declarative and spoofable. Streaming decompression with a byte counter would be the real fix, but it's significant work.

## On what remains

**Before shipping:**
- Nothing blocking. The core hardening is in place and tests pass.

**Nice to have (sequenced):**
1. Streaming decompression enforcement (WI-006) — would close the header-spoofing gap. Requires wrapping `zf.open()` in a size-enforcing reader in both `_safe_extract` and `load_baseline_from_zip`.
2. CSRF test suite (WI-005) — the middleware was hardened but has no direct tests. Low risk since the tool is loopback-only, but good hygiene.
3. CLI `cmd_ask` still passes the raw question to `route_question()` without delimiter framing (unlike the web endpoint which now wraps it). Not exploitable in the CLI context (stdin, no cross-origin), but inconsistent.

## Gaps to flag

- **`ingest.py:326`** — `_check_zip_total_size` trusts `info.file_size` headers. A crafted zip with `file_size=1` bypasses the 2GB limit entirely. WI-006 filed.
- **`app.py:410-414`** — The routing LLM call wraps the question in delimiters, but `route_question()` in `narration.py` passes it as the raw user message to the LLM. If the LLM routing system prompt is ever weakened, the delimiters are the only defense.
- **`app.py:487`** — Audit event logs `sanitized` (good), but the raw `question` is still returned in the template response at line 505 (`"question": question`). If the template ever renders it without auto-escaping, XSS is possible. Currently safe because Jinja2 auto-escapes by default.
- **`tests/test_web_auth.py`** — All test clients now send `origin: http://localhost` to pass CSRF. If the CSRF middleware is ever changed to also check port, these tests will break silently (they'd pass the origin check but fail the port check).
- **`query_dispatch.py:29-31`** — `_PARAM_VALIDATORS` only covers `settings_at_som`. If new queries with params are added, they need entries in all three tables. The architecture test catches missing _QUERY_DISPATCH entries but not missing _PARAM_VALIDATORS entries for new queries with typed params.
