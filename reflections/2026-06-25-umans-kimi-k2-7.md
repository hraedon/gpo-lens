**Work summary:** Stood up the gpo-lens web UI against an WORK-DOMAIN.local sample export and walked through the dashboard, inventory, GPO detail, conflicts, directory, ask, ingest, and baseline/resultant views. ## On the project

The web layer is in good shape for a security tool: FastAPI/Jinja2, minimal JS, and a consistent dark-themed design. The biggest risk is the same one that keeps showing up in this codebase — display text that bundles semantic meaning into terse labels that don't say what they mean. "User off" sounds like a runtime outage, not a disabled GPO side; "Overrides" sounds directional in a precedence context where the reader really wants to know "who lost." These are easy to fix but easy to miss, because the original author obviously knew what they meant.

The data model is rich enough that the UI can expose a lot, but the templates sometimes under-count columns or over-fit wide screens. The `gp-table-wrap` responsive scroll is gated behind a small-screen media query, so desktop users with long registry identities just see the right-hand columns disappear without a scrollbar. That's a recurring pattern, not a one-off.

## On the work done

The wording changes are low-risk and clearly better. I verified each in the live UI and ran the full test suite afterward — only one snapshot assertion needed updating (the Ask empty-state header). The conflicts page rename is especially defensible: the column was already showing the losing GPOs with their values and deep links; it just needed a header that says so.

The dashboard findings-count fix is the change I'd most want reviewed. Switching from `findings|length` (the paginated page) to `filtered_findings_count` corrects a real under-counting bug in the header pill, but it also means the "No findings" state now reflects the full filtered set. That's the correct behavior, but any downstream assertion that depends on the exact number shown by the pill could change. No tests broke.

I considered enabling the Ask form for deterministic queries when narration is unavailable, but that would have required route and permission changes well beyond the small wording fixes. The updated copy is honest about the limitation now.

## On what remains

1. **Horizontal clipping of wide tables (WI-053)** — the user's screenshot confirms this isn't just theoretical. The fix is likely moving `overflow-x: auto` out of the small-screen media query and/or adding `min-width`/max-width constraints so the Setting column doesn't swallow the viewport. I filed a breadcrumb rather than fixing it because the user wants to retest on their version first.
2. **WI-052 (misleading `APPLY_RIGHTS` name)** — still open. I only softened the symptom in the UI via a tooltip on "Apply Group Policy" delegation rows; the constant name and raw SDDL display in danger findings still need work.
3. **Other usability notes from subagent review** — per-page controls, deep-linking SOM names from GPO Links, Ask deterministic fallback, and nav active-state ARIA were surfaced but not acted on. Most are small and can be picked off individually.

## Gaps to flag

- **`src/gpo_lens/web/templates/dashboard.html:9`** — header pill now uses `filtered_findings_count`. Correct, but verify against estates where the total count is large and the filter hides info findings by default; the pill correctly flips from the severity-based label to a count label.
- **`src/gpo_lens/web/templates/conflicts.html:49`** — the renamed column still shows the same data as before. If the user expects a *separate* column with just loser names, this change doesn't do that; it's a relabel, not a new column.
- **`tokens.css:295,447`** — `.gp-table-wrap` has `overflow: hidden` by default and only scrolls on small screens. This is the root cause of columns disappearing on desktop when settings names are long.
- **No automated UI/visual regression tests** — these template changes were verified by manual browser screenshots. There's currently nothing that would fail if a header text regresses.
