## On the project

The four-agent review pattern is the right tool for a "are we done?" gate. Two
adversarial reviewers independently flagged the disabled-side charter violation
from different angles — that convergence is what raised my confidence enough to
treat it as a bug rather than a design choice. The most important finding
wasn't a feature gap; it was that `settings_at_som` was silently violating the
project's own charter ("flag, don't simulate") by presenting disabled-side
settings as effective. That's the exact failure mode the tool exists to
prevent.

## On the work done

**What I'm confident in:** the disabled-side fix (skip `from_disabled_side`
settings in `_fold_chain_to_buckets`) is correct and charter-compliant; the
fixture test that had been asserting the *bug* is now a positive test of the
fix. The SID-prefix tightening (`s-1-5-21-` required before matching `-515`) is
a strict improvement — RID 515 in a domain is always Domain Computers, so the
prefix is both necessary and sufficient. The determinism pass (sorted
`settings_dump`/`conflicts`/`som_conflicts`/`precedence_conflicts` + `ORDER BY`
in `load_estate`) makes snapshot diffs reproducible.

**What I'd want a second pair of eyes on:** the `Registry.pol` identity join. I
chose to resolve blocked extensions by *replacing* the blocked placeholder
with the parsed records (`source_state="registry_pol"`), not by merging with
report-rendered settings. The report uses hive-prefixed key paths
(`HKLM\Software\...`) while `Registry.pol` uses bare paths
(`Software\Policies\...`), so a dedup-by-identity merge would be fragile. The
replace-on-blocked approach sidesteps that, but means a GPO whose report
rendered *some* registry settings AND has a blocked extension won't get the
report-plus-pol union. Acceptable for v1 but worth noting.

**What was awkward:** the parallel-session overlap. Arriving at /end to find
the tree already committed, pushed, and reflected by a same-day same-model
session was disorienting. The lesson for the harness: when two sessions touch
the same repo same-day, the second `/end` should verify its changes survived
into HEAD rather than assume an uncommitted diff.

## On what remains

**Decisions the user still owes:**
- **Gap 5 alternatives.** I proposed three in-charter substitutes for the
  deferred STIG/multi-domain/RSoP items: (a) a bring-your-own-mapping
  compliance projection over the existing ADMX crosswalk, (b) a cross-DB
  estate index without changing the single-domain model, (c) a
  machine-readable "what to verify manually" caveat checklist. My
  recommendation was (a) as the highest-value, most-aligned additive surface.
  User has not yet chosen.
- **WI-020 (filed):** the frozen JSON contract is only half-pinned — ~7 of
  ~30 `--json`-emitting commands have golden tests. Decide: pin all, or mark
  the rest "informational" and refuse `--json` on them.
- **WI-021 (filed, high):** web auth grants full admin when
  `GPO_LENS_AUTH_TOKEN` is unset; the loopback-only assumption is enforced only
  at the CLI. This is the most important open security item.

## Gaps to flag

- **WI-021** — web auth default. The permission system is a facade in the
  default config; a reverse proxy or direct `uvicorn` invocation bypasses the
  CLI's loopback guard entirely. Not fixed — policy call (deny-by-default vs
  random startup token vs per-request peer check).
- **WI-020** — JSON contract completeness. A consumer building on
  `show --json` or `settings-diff --json` is on undocumented ground.
- **`query_dispatch.py` doc claim** — AGENTS.md calls it "single source of
  truth for CLI and web" but only `ask` uses it; every other CLI command calls
  `queries` directly. Minor doc/code drift; not filed separately.
- **Performance at scale** — `gpo_by_id` is O(n) linear scan; `baseline_diff`
  and `excessive_writers` have O(n²) display-name resolution. Fine at ~130
  GPOs (2.18s ingest per the parallel session), will hurt at 10k+. Not blocking
  v1.0.
