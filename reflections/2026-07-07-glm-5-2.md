# Session Reflection — 2026-07-07

**Work summary:** Diagnosed and fixed two issues reported by the user: (1) a
404 on every GPO detail page for old estates, caused by commit cb21237's
canonical_guid change (hyphen-stripping) not being reflected in already-stored
DB rows; (2) added principal-name lookup (not just SID) to the Resultant form.
Both shipped in commit d9c7192, then folded into the v1.0.0 release (ad94660).

---

## On the project

gpo-lens is in good shape post-1.0.0. The charter — deterministic core, no AI
in the truth path — is well-enforced by the import-boundary test and the module
map in AGENTS.md is accurate enough to navigate by. The one fragility I keep
bumping into is the canonical_guid contract: it's load-bearing across ingest,
store, topology, and the web layer, but it changed semantics (hyphen-stripping)
in a "fix" commit (cb21237) without a DB migration or a backward-compat shim.
The dual-key fix in gpo_index papers over it, but the lesson — that
canonical-form changes need migration planning — is worth remembering.

## On the work done

The 404 root-cause was satisfying to nail: git log -S on normalize.py showed
the exact line that changed (`return cleaned.lower()` → `return bare.lower()`)
and the commit (cb21237) that introduced it. The fix went into
`Estate.gpo_index` (dual-key: stored id + hyphen-stripped form) rather than
the store load path, which was the right call — it's one choke point instead
of six row-decoding sites, and it doesn't require touching frozen
dataclasses. I'm confident in this; the test
`test_gpo_by_id_finds_hyphenated_via_canonical_lookup` exercises both
directions.

The principal-name improvement reused `_build_name_to_sid_map` which already
existed in merge.py for delegation resolution. The new `resolve_principal_input`
is a thin wrapper. I'm less confident about the computer_sid path — I apply
the same resolution, but I didn't add a test for resolving a computer *name*
(as opposed to a user name or SID). The SAM-name resolution should work the
same way, but it's unexercised.

## On what remains

- **Regista migration 38 is pending** for the gpo_lens schema. I couldn't file
  breadcrumbs for either fix because `agent-notes breadcrumb file` fails with
  MIGRATION_REQUIRED. Someone needs to run the migration before breadcrumbs
  will work again for this project.
- **No breadcrumb filed** for the 404 fix or the principal-name improvement.
  Both should be tracked once regista is unblocked.
- The `gpo_names` property still keys by `g.id` only (not dual-keyed). It works
  today because all callers look up with `link.gpo_id` from the same DB, but
  it's inconsistent with the now-dual-keyed `gpo_index`. Low priority.

## Gaps to flag

- **regista migration drift** — `agent-notes` is non-functional for gpo-lens
  until migration 38 is applied. This blocks breadcrumb filing and the
  reconcile workflow.
- **canonical_guid has no test pinning its output format.** The cb21237 change
  shifted the canonical form from hyphenated to non-hyphenated, and nothing in
  the test suite caught it because the tests were written after the change. A
  golden test asserting `canonical_guid("{31B2F340-016D-11D2-945F-00C04FB984F9}")`
  == a specific literal would have caught the regression at commit time.
- **computer_sid name resolution is untested.** `resolve_principal_input` is
  tested for user names and SIDs, but the route handler also passes
  `computer_sid` through it and there's no test for that path
  (`resultant.py:81`).
