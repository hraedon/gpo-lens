## On the project

The project is in a strong position. v0.8.0 shipped yesterday with ~2000
tests, clean lint/mypy --strict, and the spec-driven approach (AC-numbered
specs, architecture boundary tests, identifier gate) has held up through
46 reflections across multiple models. The codebase reads well — the
separation between deterministic core and AI narration layer is clean,
and the import boundary is enforced by test, not convention.

The merge module (`merge.py`) is the most complex piece. It's 1126 lines
now, and the `principal_resultant` function alone is ~200 lines with the
loopback branches. The function is doing a lot: token building, chain
resolution, security gate evaluation, loopback detection, chain collection
with different token sets per side, merge resolution, exclusion
deduplication, conditional dangers, and caveat building. It's correct and
tested, but it's approaching the point where a refactor into smaller
composable functions would help readability. The existing helper functions
(`_collect_chain_entries`, `_evaluate_security_gate`, `_build_caveat_*`)
are the right pattern — the loopback logic could benefit from being
extracted into a similar helper.

The adversarial review process is working well. Both reviewers found the
same two real bugs independently, which gives confidence the review is
catching genuine issues rather than noise. The bugs were small (one-line
fixes) but would have caused real correctness issues in production.

## On the work done

The deny-ACE fix was the simpler change and the one I'm most confident in.
The core insight — checking `deny_sids & token_sids` before `allow_sids &
token_sids` — is obviously correct. The SDDL deny-only path fix (populating
`deny_sids` for deny ACEs with read/apply rights but no corresponding allow)
was the trickier part, and the test `test_sddl_deny_only_ace_blocks_gpo`
covers it well.

The loopback implementation is more complex and the one I'd want a second
pair of eyes on. The replace mode is straightforward — user chain ignored
for User-side, comp token for User-side security gate. The merge mode has
a subtlety I'm less sure about: the order offset (`max_user_order + 1`)
ensures comp-chain entries strictly win, but I'm relying on `max()` picking
the first entry on ties for the non-offset case, which is an implementation
detail of CPython's sort stability rather than a guaranteed contract.

The exclusion deduplication was the messiest part. In loopback mode, the
same GPO can be excluded from User-side (comp token fail) and included in
Computer-side (combined token pass). The dedup by `(gpo_id, kind)` is
correct but the semantic shift (excluded and settings are no longer
disjoint) is documented in AC-15 but not enforced by any test. I filed
WI-078 for this.

I'm confident in the test coverage for the happy paths. The edge cases
(disabled link, mixed mode, offset at order 0) are all covered by the
regression tests added after the review. What's missing is a test for the
excluded/settings overlap itself — I should have added one but the /end
process is already running.

## Gaps to flag

- **Excluded/settings disjointness broken in loopback mode** (WI-078). A
  GPO can appear in both `PrincipalResultant.excluded` and
  `PrincipalResultant.settings` when it fails the security gate for one
  side but passes for another. Documented in AC-15 but not tested.

- **`_build_conditional_dangers` treats partially-applied GPOs as fully
  gated** (merge.py:747-784). A GPO that contributes Computer-side settings
  but was excluded from User-side will still have its dangers surfaced as
  "conditional." This is misleading — the GPO is partially applied, not
  fully gated. Requires the `ExcludedGpo.side` field from WI-078 to fix
  properly.

- **`som_effective_gpos` parent-OU walk** (WI-076). All chain-based queries
  miss inherited GPOs when the target SOM is a child OU. In practice most
  estates link at the domain root, so this rarely bites — but it's a silent
  correctness gap.

- **Pre-existing uncommitted changes** in CHANGELOG.md, pyproject.toml,
  web/*, cli/* (9 files). These appear to be from prior sessions that ended
  without committing. Should be investigated before the next commit to
  avoid sweeping unrelated changes into the deny-ACE/loopback commit.

- **`max()` tie-breaking is CPython implementation detail**. In
  `_merge_bucket`, `max(items, key=lambda it: it.order)` returns the first
  entry on ties. The merge-mode offset (`max_user_order + 1`) prevents
  ties in the normal case, but the non-loopback path and the offset=0 case
  (empty user_entries) rely on insertion order. This is stable in CPython
  3.7+ but not guaranteed by the language spec.
