## On the project

The codebase continues to improve. The previous session's audit (WI-045 through WI-051) correctly identified the main weakness: the same conceptual operations were reimplemented in each module with subtle drift. This session's consolidation work (WI-046/047) pushed `authz.py` further as the single source of truth for authorization predicates. The `iter_sddl_apply_aces` function and `SddlApplyAce` dataclass are clean abstractions that eliminated the most dangerous duplication — the SDDL-fallback pattern that existed in topology.py, danger.py, and merge.py with different filtering logic.

The `Estate.gpo_index` / `gpo_names` properties (WI-045) are a high-value, low-risk improvement. Every call site that previously built `{g.id: g for g in estate.gpos}` or `{g.id: g.name for g in estate.gpos}` now uses the cached property. The lazy cache is safe because no production code mutates `estate.gpos` after construction.

The `SettingRaw` TypedDict (WI-048) is a partial win. `element_to_dict` now returns a typed value, but `Setting.raw` remains `dict[str, object]` because `registry_pol` creates dicts with non-standard keys. The `cast(dict[str, object], raw)` in ingest.py is a pragmatic workaround — at runtime `SettingRaw` IS a dict, so the cast is a no-op. A future sprint could unify the registry_pol raw dict structure with the `SettingRaw` contract.

## On the work done

The consolidation work is solid. The GLM adversarial reviewer caught a real bug: `_grants_read_or_apply` in topology.py's `is_security_filtered` was using substring matching (`"read" in p or "apply" in p`) which missed "Edit settings" — a permission that DOES include Read per Microsoft's GPOGroupedAccessEnum spec. This created an internal contradiction: `is_security_filtered` said a GPO was filtered, while `security_filtering_detail.has_au_read` said AU had read. The fix (replacing `_grants_read_or_apply` with `permission_implies_read`) changed the test expectations for two tests that encoded the old behavior. I updated both tests and verified the new behavior is consistent across all three checks (is_security_filtered, has_au_read, ms16_072_vulnerable).

The Kimi reviewer caught another important issue: `store.py`'s `load_estate` still used post-construction Gpo mutation (the same pattern WI-050 was meant to eliminate in `_parse_gpo`). I refactored it to collect links/settings/delegation per-GPO first, then construct each Gpo with all data — consistent with the new construction invariant.

WI-049 (`_safe_json_loads` now raises instead of silently swallowing) is a deliberate behavioral change aligned with the coverage-honesty charter. The two tests that previously expected silent degradation were updated to expect `json.JSONDecodeError`. This is the right call — presenting a partial estate as complete is worse than failing to load.

WI-051 (dedup prevention at source) is clean. The `_already_excluded` set is passed to both chain-collection calls, and the second call skips GPOs already excluded from the first. The reviewers confirmed this is correct because security filtering is per-GPO (not per-side).

## Gaps to flag

- **`cast(dict[str, object], raw)` in ingest.py:532,556** — Runtime no-op but masks the type mismatch between `SettingRaw` (TypedDict) and `Setting.raw` (`dict[str, object]`). If a future change makes `Setting.raw` stricter, these casts would silently suppress type errors.
- **`is_default_writer` name-spoofing surface** — The expanded name set (now includes "Creator Owner", "Owner Rights", etc.) is checked by display name in queries/_delegation.py, not by SID. A maliciously named group could evade the non-default-writer flag. Low risk in practice (GPO delegation data comes from AD, not user input).
- **Merge.py SDDL fallback still inline** — WI-046 left merge.py's SDDL fallback as-is because it needs deny-ACE tracking. This is correct but means the "three different permission-checking strategies" issue is only partially resolved. Merge.py's delegation path now uses `permission_implies_read`, but its SDDL path still has inline rights parsing.
- **`load_estate` error handling** — WI-049 makes corrupt JSON raise `json.JSONDecodeError`, which aborts the entire snapshot load. There's no per-row recovery. This is intentional per the charter, but CLI/web callers should surface a clear error message to the operator.
