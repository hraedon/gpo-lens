## On the project

The Plan 023 findings work is a genuinely good direction and the plans that
accompany it (024/025/026, all *Proposed*) are unusually rigorous — 024 in
particular reads like a careful critique of the v0.1 that shipped alongside it.
The important thing for the next team member to internalise: **`findings.py` as
committed is a v0.1 that Plan 024 supersedes.** Build toward 024's model
(separate finding-definition / evaluation-run / occurrence / observation /
triage-event, coverage-aware resolution, canonical fingerprints, least-priv
triage perms) rather than piling more onto the current crude version.

The philosophical shift is real and now accepted: gpo-lens holds durable,
operator-authored triage state — it is no longer a pure function of its exports.
That is defensible because triage is append-only and provenance-tracked, but it
is a change worth being conscious of.

## On the work done

**The good:**
- **Coverage-safety fix (the one that mattered).** `update_finding_lifecycle`
  marked *any* active finding absent from a scan as `resolved`, with no notion of
  collection completeness. On this estate — where denied SYSVOL subtrees and
  per-GPO collection errors are normal and expected — a partial re-collection
  would have reported live cpassword / MS16-072 findings as "resolved." That is
  the worst failure direction for a posture tool. Added `_absence_is_meaningful`:
  under complete coverage behaviour is unchanged; under a partial collection,
  findings whose subject GPO was not actually re-evaluated stay active and are
  counted `indeterminate`. Ingest passes real coverage
  (`coverage_complete=not estate.coverage_gaps`). This is Plan 024 §7.5 pulled
  forward because it is a safety issue on a load-bearing tool. +4 regression
  tests.
- **E2E verified**, not just unit-tested. Drove the exact ingest lifecycle block
  over the real 14-GPO fixture estate: 23 findings populate across real
  categories, and `GET /findings` renders them (200, 43 KB). The ingest path is
  wrapped in try/except-log, so a silent failure there would never surface in a
  test — worth re-checking whenever that block changes.
- **Test speed.** Added `pytest-xdist`; local runs are now parallel (`-n auto`)
  and coverage-free by default (coverage roughly doubles wall time: ~97s → ~170s
  measured; the 85% floor was already CI-only). Full suite ~175s → ~55–97s
  locally. CI re-adds `--cov` explicitly.

**The not-so-good / traps I hit:**
- **`COVERAGE_CORE=sysmon` hangs under xdist** with coverage 7.14 (controller
  sits at ~0% CPU indefinitely). Looked like the obvious win; measured it, it
  hangs. Do not wire it in without re-testing on a newer coverage.
- The reflection lore that "CI times out at 120s" is **not true** — there is no
  `timeout-minutes` in ci.yml (GitHub default is 360 min). The suite was slow,
  not timing out.
- I twice reported a suite as "passing" off a pipeline whose exit code was
  `tail`'s, not pytest's. Check `PIPESTATUS`/`$?` of the actual test process, not
  the tail.

## Gaps to flag

- `src/gpo_lens/web/routes/ingest.py` — the finding-lifecycle update is wrapped
  in `except Exception: log.warning(...)`. Correct for resilience (a detector
  crash must not fail ingest), but it means findings can silently fail to update.
  Any change to that block needs the e2e check above, not just unit tests.
- `src/gpo_lens/findings.py:_finding_to_key_parts` — `detail or summary` is a
  weak dedup discriminator; two genuinely distinct findings on one GPO with the
  same summary still collide. Plan 024 §4 (declared distinguishing dimensions) is
  the real fix.
- The `finding` table is not scoped by estate/domain. A single store is assumed;
  ingesting two different estates into one DB would make each resolve the other's
  findings. Fine today, undocumented.
