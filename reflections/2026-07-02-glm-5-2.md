## On the project

The codebase is mature and well-structured. The design system in `tokens.css` is genuinely good — the token-based approach meant the 11px→12px change was a one-line edit to `--fs-xs` plus a sweep of hardcoded values to `var(--fs-xs)`. The "instrument panel" aesthetic is coherent and the CSS is clean enough to extend without fighting it.

The route handler pattern (all `async def` doing sync SQLite) was the single highest-value fix. It's a classic FastAPI footgun — the framework doesn't warn you, and in dev with one user it's invisible. The fix is mechanical but touches 16 files, which is why it hadn't been done. The subagent handled it well but missed `baseline_post` and `golden_diff_post` needing `asyncio.to_thread` wrapping (they stayed async but their sync ops weren't offloaded). I caught that in review.

The Plan 022 doc is well-written — each WI has explicit ACs and the sequencing rationale is sound. The plan correctly identifies WI-1 as "the only one a deployed user can feel."

## On the work done

**What went well:** The parallelization strategy worked cleanly. I dispatched two subagents (WI-1 and WI-8) in parallel while handling WI-9 and the small frictions myself. The file partitioning was clean — WI-1 touched only `.py` route files, WI-8 touched only `danger_list.html` + `tokens.css`, and I handled `ou_detail.html` + remaining templates/CSS. Zero merge conflicts.

**What I'm confident in:** The async→sync conversion is correct — FastAPI's TestClient handles both transparently, and all 2209 tests pass. The danger card grouping via Jinja2 `groupby('check_id')` is correct because `danger_findings()` sorts by `(severity, check_id, gpo_id)`, guaranteeing consecutive same-check_id findings. The `asyncio.to_thread` closures in upload handlers create SQLite connections inside the worker thread (the subagent's key insight — SQLite connections can't cross threads).

**What I'd want a second pair of eyes on:** The `_compute_diff` closures in `baseline_post` and `golden_diff_post` capture `baseline_gpos`/`golden_gpos` from the enclosing scope. These are loaded inside the `TemporaryDirectory` context manager, but they're fully in-memory Python objects before the context exits, so the temp dir cleanup doesn't affect them. I'm confident this is correct but it's the kind of thing that deserves a careful read.

**The adversarial review was valuable.** The Kimi K2.7 reviewer found 4 real issues I'd missed: 3 CSS rules still hardcoding `11px` (`.gp-badge`, `.gp-applies-all`, `.gp-tab .count`), the fragile string matching for site-links detection (fixed to use `topology.has_site_links()`), missing test coverage for the caveat summary, and hardcoded `12px` in `conflicts.html`/`trends.html`. All low severity, but fixing them made the change set complete.

## Gaps to flag

- **`ask_post` handler does sync LLM calls.** It's now `def` (runs in threadpool), which is correct — but a slow LLM response still occupies a threadpool slot. If the threadpool fills, other requests queue. Not a bug, but worth noting for capacity planning if narration gets heavy use.
- **`test_ui_regression.py` has 233 tests but only 2 cover OU detail rendering.** The new caveat summary test I added is minimal — it checks that the summary element exists but doesn't verify specific counts against the fixture. A stronger test would assert exact mechanism counts for a known OU.
- **The `gp-table` class is still used on many pages** (OU detail settings, conflicts, changelog, GPO detail). The danger page is the only one that moved to cards. If the card pattern proves better, it could be applied to other dense tables — but that's a design decision, not a gap.
- **`--fs-xs` is now 12px, but `.gp-eyebrow` (10.5px) and `.gp-ver` (10.5px) are still hardcoded.** These are display elements (labels, version tags), not body text, so 10.5px is defensible. But if someone bumps `--fs-xs` again, they won't know to check these.
