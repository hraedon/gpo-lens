## On the project

gpo-lens continues to be the most architecturally disciplined project in the portfolio. The charter constraints ("no AI in the truth path," "flag don't simulate," "coverage honesty") are not aspirational — they're enforced by architecture tests, and this session's work made the deterministic core *more* honest without breaking that constraint. The zip-permission fix (`_scan_sysvol_coverage`) is a direct expression of the coverage-honesty principle: when GPP content is invisible due to lost traversal bits, the tool must say so, not silently present an incomplete estate as complete.

The golden estate fixture (WI-011) was the highest-leverage work item. The prior flat-only fixture hid two shipped-green-but-broken bugs. The new fixture with real nested SYSVOL layout (uppercase MACHINE/USER, per-CSE subfolders, V2 tasks, cpassword, Blocked Registry.pol, drive mappings, ILT) is the CI gate that prevents the next silent regression. The scale test against the real 129-GPO estate validated that the fixture's shape matches reality.

The multi-agent dispatch pattern worked well with one important refinement: strict file-ownership boundaries. The prior session's reflection warned about a near-miss where kimi reverted glm's ingest.py changes. This session avoided that by partitioning files (Track A: detection/ingest/registry_pol/admx_parser; Track B: topology tests/docs/CLI) and sequencing the golden fixture work (Track C) after the parser fixes landed. Zero conflicts.

## On the work done

**What I'm confident in:**
- The `_walk_gpp_xml` refactor to yield `_GppXmlFile(side, cse, tree, rel_file)` is clean. All callers updated, no raw path manipulation remains. The deduplication fix (mixed flat+nested layouts) was caught by the adversarial reviewer and is correct — prefer nested paths by component count.
- The V2 task parsing (nested `<Exec><Command>`) works on real data. Both estates have V2 tasks with commands correctly extracted. The golden fixture's TaskV2 and ImmediateTaskV2 test both paths.
- The `_scan_sysvol_coverage` coverage-honesty fix is the most impactful change. On a fresh zip extraction of the 129-GPO estate, it detects 44 `unreadable_sysvol` coverage gaps with actionable `chmod -R +rX` remediation guidance. This is charter-compliant and user-friendly.
- The golden estate is well-scrubbed: synthetic GUIDs, GOLDEN.local domain, no real identifiers. The PReg binary is byte-correct (verified by adversarial reviewer against the spec).

**What I'd want a second pair of eyes on:**
- The `_scan_gpp_xml_for_refs` Properties-child fix changes ref_type classification: Drive/Printer SYSVOL refs are now `drive_mapping_unc` instead of `gpp_file_ref`. This is more correct but changes the doctor output for estates where both SYSVOL and settings find the same UNC. The dedup logic handles this (first-found wins on equal rank), but the semantic shift is worth verifying against the real estate's 424 drive-mapping UNC refs.
- The `_scan_sysvol_coverage` function tries to `list(entry.iterdir())` on each CSE subdirectory to detect unreadability. This is an O(n) stat per subdir — negligible for 129 GPOs (2.18s total ingest) but worth profiling against larger estates.
- The golden estate's AllGPOs.xml for the drives GPO includes Drive Maps ExtensionData, but the V2 task GPO's report only includes the V1 task (V2 is in SYSVOL only). This mirrors real GPMC output but is undocumented in the fixture generator.

**What went well:** The adversarial review pattern (2 independent reviewers) caught a CRITICAL bug (deduplication on mixed layouts) that both implementation agents missed. The scale test against real data found two bugs (zip permissions, Properties-child UNC scanning) that neither the implementation agents nor the adversarial reviewers caught — because synthetic fixtures can't expose real-world extraction issues. The lesson: always validate against real data, not just synthetic fixtures.

## Gaps to flag

- **`detection.py:155-176`** — `_GPP_PATH_ATTRS` still checks attributes on the OUTER GPP element (e.g., `<Drive Path="...">`) in addition to the new Properties-child check. On real GPP XML, the outer element never carries these attributes — the Properties-child check is what actually finds them. The outer-element entries are dead code that could be removed for clarity, but leaving them is harmless (defense-in-depth).
- **`ingest.py:898-918`** — `_scan_sysvol_coverage` calls `ci_child` for each side dir, then `prefs.iterdir()`, then `entry.iterdir()` per CSE subdir. On the 129-GPO estate this is fast (part of the 2.18s ingest), but it's three levels of filesystem traversal per GPO. A `os.scandir` fast path would help on larger estates.
- **`test_golden_estate.py`** — The `test_v2_task_uses_nested_layout` test asserts exact path equality (`MACHINE/Preferences/ScheduledTasks/ScheduledTasks.xml`), but the casing depends on what the generator writes. If the generator ever changes casing, the test breaks. This is correct behavior (the test should pin casing), but the dependency is implicit.
- **Real-estate validation gap** — The scale test ran detection scans against the 129-GPO estate but didn't validate the doctor findings against ground truth (e.g., are the 424 drive-mapping UNC refs actually broken? Are the 112 MS16-072 vulnerabilities real?). The tool's accuracy on real data is inferred from the synthetic test suite, not independently verified.
- **WI-018 decision needed** — The FastAPI web app is significant maintenance surface. The user should decide whether to keep investing in it or freeze it. The CLI + HTML report path covers the deployable no-AI pitch without the web overhead.
