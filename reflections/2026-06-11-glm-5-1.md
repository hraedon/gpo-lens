## On the project

gpo-lens is a genuinely well-structured project. The deterministic core (stdlib-only, no AI in the truth path) is clean and the import boundaries are enforced by architecture tests. The optional-subsystem pattern (narration first, now web) works — `pip install -e .` without extras stays lean, and the web extra adds FastAPI without polluting the core.

The plan (012) was unusually detailed and honest — it named specific cert-watch lessons, pre-declined entire categories (in-app credentials, per-OU scoping, multi-tenancy), and gave explicit trigger-gated follow-ons. This made execution straightforward: the plan said what to build, why, and what not to build. More plans should be written this way.

The fixture system (`tests/fixtures/` with a synthetic estate generator) is excellent for this kind of multi-agent work — every agent could validate against the same 10-GPO synthetic estate without needing the gitignored real samples.

## On the work done

**What went well:**
- Parallel agent dispatch worked. GLM and KIMI produced working code simultaneously on independent work items, roughly halving wall-clock time per phase.
- Two adversarial reviewers per phase caught different bugs. Reviewer 1 found the zip-slip `startswith` bypass; Reviewer 2 found the missing baseline size cap. Neither caught both independently.
- The wave pattern (implement → review → fix) prevented compounding. Fixing EV issues before W0, and W0 issues before W1, meant each layer was built on corrected foundations.
- 526 tests pass, mypy strict clean on 32 files. The test-to-code ratio is healthy.

**What was awkward:**
- Four agents all editing `app.py` caused merge conflicts that fix agents had to untangle. The work-item boundary does not align with the file boundary.
- The `_QUERY_DISPATCH` table was copy-pasted from `cli/_narration.py` into `web/app.py`, creating a drift risk identical to WI-003. A shared module should have been extracted during W0, not left as reviewer feedback.
- The `override_principal` hack (mutating a frozen dataclass via `object.__setattr__`) was a code smell that survived through two rounds of fixes before being replaced with proper `dependency_overrides`.
- Ruff issues accumulated across waves. Should have linted between each wave, not just at the end.

**Confidence:**
- High confidence in the event store (EV), sinks, permission seams, and architecture boundaries.
- Medium confidence in the ask/narration view — it duplicates the CLI's query dispatch and relies on `route_question` which itself calls the LLM for routing. The prompt injection surface is acknowledged but unmitigated.
- The zip-slip fix (`is_relative_to`) is correct but there's a remaining symlink-through-zip gap (flagged by a reviewer but not fixed — see Gaps).

## On what remains

**Before this can ship (v0.3.x / v0.4.0 per the plan's release framing):**
1. Extract `_QUERY_DISPATCH` into a shared module (narration or a new `narration_dispatch.py`) to close the WI-003 drift risk on the web side.
2. Update `CHANGELOG.md` with Phase EV and web UI entries.
3. Consider streaming the upload read (currently `file.read()` buffers the full payload before the size check) for production hardening.
4. Add zip-slip test (zip with `../` member) and size-cap test (413 response for oversized upload).

**Nice to have:**
- The web's `_QUERY_DISPATCH` should have a drift guard test matching `test_query_dispatch_matches_valid_queries`.
- CSRF protection on POST forms (acceptable to skip for loopback-only, but worth noting).
- Symlink protection in `_safe_extract` (filter members with `external_attr` indicating symlink).
- The `ask` view logs the user's raw question in audit events — consider truncating sensitive questions.

## Gaps to flag

- **`src/gpo_lens/web/app.py` — duplicated `_QUERY_DISPATCH`** (~line 42-78): byte-for-byte copy of `cli/_narration.py`'s dispatch with no drift guard. If someone adds a query to `_VALID_QUERIES` and updates only the CLI dispatch, the web silently returns "not implemented."
- **`src/gpo_lens/web/app.py` — symlink zip-slip** (`_safe_extract`, ~line 33): `is_relative_to` blocks `../` traversal but `extractall` creates symlinks. A zip with a symlink pointing outside the dest passes the check.
- **`src/gpo_lens/web/app.py` — upload memory DoS** (~line 271, ~line 499): `await file.read()` buffers the entire upload before the size check. A streaming read with early abort would be safer.
- **`src/gpo_lens/web/app.py` — prompt injection** (~line 408): user question is interpolated directly into the LLM prompt with no sanitization. Local-only binding mitigates but doesn't eliminate the risk.
- **`tests/test_web.py` — no zip-slip test**: the `_safe_extract` function has no test that crafts a malicious zip with `../` paths.
- **`tests/test_web.py` — no size-cap test**: the 413 response path for uploads is untested.
- **`plans/012-local-web-ui.md` — plan document committed**: contains detailed architectural decisions; should be reviewed by the user before the next session builds on it.
