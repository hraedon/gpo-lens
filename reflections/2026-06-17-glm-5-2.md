## On the project

The "deterministic core / flag-don't-simulate" charter is genuinely
institutionalized here, not aspirational — the import-boundary architecture
test, the tier separation, and the coverage-honesty reconciliation
(gpo-inventory + collection-errors + export → named CoverageGaps) are real and
load-bearing. That's rare; most "no AI in the truth path" projects leak
somewhere, and I couldn't find a leak.

The weak spots are consistent with a project that grew feature-first: **caveat
surfacing was honest in the CLI but lagged in the secondary outputs** (report
HTML/MD and the web OU-detail page both rendered "effective settings" without
the scoping caveats — the most material charter-integrity gap, now fixed), and
**contract-freeze discipline existed but was half-applied** (the `--json`
envelope test pinned 11 of ~39 commands). The large modules (`queries.py` 1455
lines, `detection.py`) are god-ish but defensible for a single-domain tool; the
real latent risk was the **duplicated SDDL/broad-trustee security predicates**
between detection and topology — exactly the kind of drift that silently
corrupts a security finding or a scope flag for thousands of GPOs. That's now
unified in `authz.py` with the two predicates kept deliberately distinct.

## On the work done

The player-coach split worked. Disjoint file ownership let 4 subagents run in
parallel each round with zero merge conflicts, and running only each agent's
own verification scope (plus a full-suite gate at the end) caught the one
integration fallout — `baseline_diff` corpus rows that the de-routing agent
correctly predicted but couldn't own — before commit.

Most confident in: the report/web caveat surfacing (charter-critical, small,
well-tested), the contract freeze (mechanical but now actually enforceable),
and the `authz.py` extraction. The **M3 SDDL unification** was the riskiest;
the "write golden tests first, keep the two predicates distinct, only unify the
substrate" discipline is what made it safe, and the genuine semantic
differences (Everyone is broad for scope but ignored by MS16-072; an
AU-allow+AU-deny pair is filtered but not MS16-072-vulnerable) are now
documented and locked by tests rather than accidentally merged.

I'd want a second pair of eyes on two things. (1) The **M5 decision** to fully
remove `baseline_diff` from LLM routing rather than allowlist a baselines
directory: I chose removal for simplicity, and it's the safer default, but it
means `/ask` can no longer answer "compare against the W11 baseline" at all —
worth confirming nobody expected that workflow via the web UI. (2) The **L5
calibration bound** (`>= 28` → `>= 30`) was edited but never run because the
test is `@pytest.mark.samples` and skips without `samples/`; low confidence the
number is exactly right until someone runs `-m samples` against the real
exports.

## Gaps to flag

- **Calibration bound unverified** — `tests/test_calibration.py::test_loopback_detected` tightened to `>= 30` but samples-gated and not executed this session. Verify under `-m samples`.
- **`baseline_diff` no longer `/ask`-able** — `src/gpo_lens/query_dispatch.py`. Confirm no web workflow relied on it; CLI `baseline-diff` is unaffected.
- **`queries.py` is still a 1455-line god-module** mixing pure-over-`Estate` queries with `sqlite3.Connection`-bound snapshot-diff. The adversarial review flagged splitting `snapshot_diff` out; not done. The risk is a caller passing the wrong object type to the wrong function.
- **`report.py` MD/HTML rendering is still duplicated per-section** (`_effective_settings_md`/`_html` etc.). I added caveats to both copies of the effective-settings section, so the duplication now has one more thing to keep in sync. A single `render_section(..., format=)` refactor would eliminate the class of bug.
- **Store migration pattern is versioned but unproven** — `store.py` now stamps `user_version` and rejects future DBs, but there is still only one real (additive) migration. The ordering/idempotency discipline won't be exercised until migration #2 lands.
- **`web/auth.py` reads the token from env on every request** — a post-startup env unset silently downgrades from token-required to loopback-viewer. Low severity (still denies non-loopback) but worth caching the token at startup.
