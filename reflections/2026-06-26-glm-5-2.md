## On the project

The codebase is in genuinely good shape. The "no AI in the truth path" charter holds cleanly: `narration.py` is the only model-touching module, the architecture test enforces the core never imports it, and the deterministic core is stdlib-only. The query_dispatch refactor was satisfying because the project had already invested in drift-guard tests (`test_query_dispatch_keys_match_query_descriptions`, `test_query_dispatch_tables_consistent`) — the single-source-of-truth refactor makes those tests *trivially* true rather than load-bearing, which is the right direction for a guard. The docs/spec discipline (`docs/spec/wi_*.md` with explicit ACs) is a real accelerator; I could scope each WI against the spec without spelunking.

One thing that struck me: the breadcrumb/WI dual-tracking surface (WI-067) had drifted into ambiguity precisely because two tracking mechanisms coexist without a documented relationship. The fix (a frontmatter `wi:` link + a triage policy in AGENTS.md) is light, but the deeper risk is that the `file-breadcrumb` *skill* lives globally in `~/.claude/skills/` while the markdown-breadcrumb convention is project-local — so agents who read only the skill won't know the project wants the frontmatter link. I patched the skill to mention it conditionally; that's a reasonable compromise but the real long-term answer is project-level skill overrides or a project-local `AGENTS.md` contract the skill reads.

## Gaps to flag

- **WI-073 (filed this session):** `call_llm`'s default model is `claude-sonnet-4-20250514` regardless of provider. When provider resolves to OpenAI and `GPO_LENS_LLM_MODEL` is unset, a true OpenAI endpoint 400s with a confusing model-not-found error. Low severity (setting the model is the documented path) but a real foot-gun on the exact code path WI-071 just fixed — worth a per-provider default or a fail-fast.
- **No integration test against a live LLM endpoint.** `tests/test_narration_integration.py` exists but still mocks transport; the OpenAI body/response shapes in `narration.py` are contract-verified, not runtime-verified. If someone wires a CI secret for a sandbox OpenAI key, an end-to-end smoke test would close the gap.
- **Spec drift on `wi_query_dispatch.md`:** the spec states `gpo_lens.merge` is imported lazily via `__import__`, but the module imports it eagerly at the top (pre-existing, not introduced this session). Now that the module docstring explicitly points at that spec, the drift is more visible — the spec or the code should be reconciled.
- **`QuerySpec.name` duplicates the registry key** with no guard that they match. I added a `spec.name == name` assertion in the strengthened test, but the field itself is redundant; a future cleanup could drop it.
- **Pre-commit identifier hook** is opt-in (`git config core.hooksPath githhooks`); the CI gate is the hard backstop. Worth confirming on fresh clones that the hook is activated, since the docs frame it as "early warning" but it's off by default.
