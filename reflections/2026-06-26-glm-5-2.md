## On the project

The codebase's clean seams continue to enable the parallel subagent workflow.
Four changes across seven files with zero file overlap between the two
implementation pairs — GLM touched CHANGELOG + ingest.py + test_ingest.py +
test_detection_resilience.py, while kimi touched test_web.py + pyproject.toml
+ ci.yml. The only shared file was CHANGELOG.md (GLM added the WI-069 entry, I
added the WI-070 entry during SHOULD-FIX application), and that was sequential,
not parallel.

The WI-070 coverage gate fix exposes a structural tension: the project's real
coverage is ~90% (with samples/) but CI runs without samples/ (gitignored),
yielding ~20.7%. The 85% threshold in `addopts` was a false-confidence hazard
that could pass locally and fail on a fresh checkout. The fix (remove from
addopts, enforce 20% in CI) is honest but 20% is a weak floor — it catches
"tests don't run at all" but not gradual erosion. The proper fix (synthetic
fixture estate so calibration tests run deterministically in CI) is a larger
body of work tracked as a future improvement.

**The GLM review of kimi's work returned empty.** This is the second time

or a model issue (GLM couldn't produce a review). I self-reviewed the kimi

**What I'm most confident in:** The WI-072 wrapper removal is clean. The
wrappers were thin filters (`[g for g in _scan_sysvol_gaps(gpos) if g.kind == ...]`)
that each called the merged walker independently, causing redundant walks when
used from tests. `load_estate` already called `_scan_sysvol_gaps` directly, so
no production code was affected. The tests now filter inline, which is more
explicit about what they're actually testing.

**What I'm least confident in:** The 20% CI coverage floor. It's the right
*type* of fix (separate local-dev from CI, be honest about the samples
dependency) but the number itself is arbitrary. If someone adds a new module
with no tests, coverage drops by maybe 1-2%, still well above 20%. The gate
won't catch that. The alternative — building a synthetic fixture estate —
is the real fix but it's non-trivial work that's been deferred for multiple
sessions now.

## Gaps to flag

- **The GLM adversarial reviewer returned empty output.** This happened on
  the kimi-implementation review. The task showed `state: completed` but
  `<task_result>` was empty. I self-reviewed the changes instead. If this is
  a recurring harness issue, the cross-model review pattern degrades to
  single-model self-review on the affected side. Worth investigating whether
  it's a timeout, a context limit, or a model-specific issue.

- **The 20% CI coverage floor is a weak regression guard.** It catches
  catastrophic failure (tests don't run) but not gradual erosion (new
  untested modules). The proper fix is the synthetic fixture estate, which
  has been deferred for multiple sessions. The gap between the 20% CI floor
  and the ~90% real coverage means CI gives a much weaker signal than local
  dev — the opposite of what a gate should do.

- **`_safe_extract` cleanup path still has an untested edge case:** the
  `dest.iterdir()` path when `dest` exists but extraction fails before any
  file is written (empty dest). The new test covers the missing-dest case;
  the existing `test_absolute_path_blocked` covers the has-files case; but
  the empty-but-existing-dest case is in between. Low risk — the iterdir
  would just return an empty iterator — but it's a gap.

- **The `test_scan_sysvol_gaps_walks_preferences_once_per_gpo` test uses
  `chmod(0o000)` to simulate an unreadable directory.** This test skips on
  Windows (chmod is a no-op there) but there's no `@pytest.mark.skipif` guard.
  It would pass vacuously on Windows CI rather than being explicitly skipped.
  The Windows pester CI job runs PowerShell tests only, not Python tests, so
  this hasn't been observed in practice — but it's a latent gap.
