## On the project

The architecture continues to be the asset. Three independent fixes in three
different modules (`narration.py`, `cli/_core.py`, `ingest.py`) with zero
file overlap, dispatched as three parallel subagents, then reviewed by three
parallel subagents — that workflow only works because the codebase has clean
seams. The `tests/_arch.py` boundary test, the `query_dispatch` table, and
the frozen JSON contract (`docs/spec/json-contract.md`) together make the
reviewable surface enumerable, which is what let the reviewers give concrete
file:line findings instead of hand-waving.

The one place where the seams *didn't* hold this session was documentation.
The WI-065 reviewer caught that `docs/spec/json-contract.md` documented the
`--json` shadowing bug as a *requirement* — three separate passages said
"post-subcommand `--json` is required." AGENTS.md says "the spec is the
contract," and the spec was lying. This is the second session in a row where
a spec doc has drifted from implementation (the prior reflection flagged the
same pattern in older reflections claiming "no CI, no CHANGELOG"). The
project needs a spec-reconciliation step somewhere — either in `/end` or as
a CI check that greps the spec for words like "required"/"must" against
actual argparse definitions. Filed mentally; not as a WI because WI-067
(process hygiene around breadcrumbs/WIs) already covers adjacent ground.

## On the work done

**The cross-model reviewer pattern with implementer-models-as-reviewers
worked.** The dedicated adversarial reviewers would have been preferable
(they're tuned for skepticism), but kimi-reviewing-glm-work and
glm-reviewing-kimi-work caught real SHOULD-FIX items in all three cases:
spec drift, stale comments, weak test assertions (the `trends` equivalence
was comparing only the key set of the first element — would have passed
vacuously), missing `help=` text, adversarial-domain regression gap,
`clear=True` test isolation, case-insensitive config matching, missing
CHANGELOG entry. None of these were BLOCKING, but every one of them would
have shipped if I'd self-reviewed. The 2x implementation cost is worth it.

What I'm least confident in after this session: the **WI-066 "union
equivalence" claim**. The implementer framed the merged walker's per-entry
`OSError` handling as a "minor superset behavior change"; the reviewer
(kimi) reframed it as "union-equivalent, not a superset" and traced through
every input class to prove it. I read both arguments and the kimi reviewer's
trace is convincing, but it's the kind of proof that's only as good as the
input classes the tracer thought to enumerate. The `_scan_sysvol_gaps` test
that asserts `iterdir` is called exactly once per subdir per GPO is the
real safety net here — if a future refactor reintroduces a two-pass design,
that test catches it. The union-equivalence argument is a one-time
correctness claim; the iterdir-count test is a permanent regression guard.

The trickiest fix was **WI-064's hostname detection**. The
`endswith(".anthropic.com")` check is load-bearing — the leading dot is
what defeats the `api.anthropic.com.evil.com` suffix attack. The
implementer got this right by accident (it's the natural way to write the
check), but the original test suite didn't pin it. The reviewer (glm)
correctly flagged "the central security property is untested" as SHOULD-FIX,
and I added a parametrized test covering four adversarial domains before
commit. Without that test, a future "simplification" to `if "anthropic" in
host:` would silently reintroduce the leak.

## Gaps to flag

- **`docs/spec/json-contract.md` was silently wrong about the `--json`
  contract for at least one release.** Three passages documented the
  shadowing bug as a requirement. Fixed in this commit, but the pattern
  (spec docs drifting from implementation) is recurring. The project needs
  either a `/end` spec-reconciliation step or a CI grep that flags
  "required"/"must" language against actual argparse definitions. Not filed
  as a WI because WI-067 covers adjacent process hygiene.

- **The `_scan_sysvol_coverage` and `_scan_corrupt_gpp_xml` wrappers still
  perform redundant walks when called independently** (e.g. from tests).
  `load_estate` is fixed, but the public-ish API surface retains the
  double-walk hazard. Filed as WI-072.

- **`call_llm`'s request body is still Anthropic-shaped regardless of
  provider.** WI-064 closed the header leak only. True OpenAI endpoints
  will get correct Bearer headers but a body they cannot parse. Filed as
  WI-071; do not claim OpenAI provider support works until that lands.

- **The cross-model reviewer pattern with implementer-models-as-reviewers
  is a stopgap, not a replacement.** The dedicated adversarial reviewers
  are usage-locked until next week. When they come back, the workflow
  should revert to implementer (kimi/glm) + dedicated-reviewer
  (nemotron-3-ultra, etc.) rather than continuing to reuse implementers as
  reviewers. The dedicated reviewers are tuned for skepticism; reusing
  implementers risks both models sharing the same blind spots on a given
  code shape.

- **`GPO_LENS_LLM_PROVIDER` silently falls through to auto-detection on
  unrecognized values.** I made the matching case-insensitive (`.strip().lower()`)
  per the reviewer, but a typo like `GPO_LENS_LLM_PROVIDER=anthrpoic` still
  silently sends auto-detected headers rather than failing. For a Tier-3
  narration layer this is defensible (graceful degradation), but it
  contradicts the fail-fast precedent `load_danger_rules` just adopted
  (WI-069). Worth a process decision: should config knobs fail-fast or
  fail-soft? Not filed as a WI — it's a policy question, not a defect.

- **The WI-066 "union equivalence" proof is only as good as the input
  classes enumerated.** The kimi reviewer traced every `OSError` path
  carefully and the new `iterdir`-count test is a strong regression guard,
  but exotic filesystems (NFS soft mounts, SELinux deny-on-stat) could
  surface edge cases neither reviewer considered. Low probability, but the
  cost of being wrong is a silent coverage gap on a security tool.

- **The prior session's reflection (`2026-06-25-glm-5-2-3`) recommended
  "build the synthetic fixture estate" as the fix for the coverage gate
  (WI-070).** That's the right fix but it's a non-trivial body of work
  (needs a synthetic GPO estate that exercises every calibration
  assertion). The cheaper fix — lower the threshold to ~21% — is honest
  about the current state but doesn't fix the underlying fragility. The
  next session should decide which path before the gate produces a false
  green signal on a real regression.
