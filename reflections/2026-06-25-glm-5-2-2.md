## On the project

The architecture continues to be the project's strongest asset. The deterministic/AI separation, import-boundary enforcement, and "flag don't simulate" topology stance are all disciplined design decisions that most security tools lack. This session's work — compliance mapping, remediation text, trend analysis, and the REST API — all fit naturally into the existing architecture without requiring any structural changes. The `query_dispatch.py` table made the REST API almost trivial to implement; the `store.py` snapshot persistence made trend analysis a thin layer over existing data.

What's most striking after this session is the gap between the tool's analytical depth (cpassword detection, MS16-072, version skew, danger rules with compliance mappings, remediation guidance, trend analysis) and its operational maturity (no CI, no tags, no CHANGELOG). The codebase has 1800+ tests and a clean architecture, but it exists only on one machine with no remote. The roadmap's Phase 0 items (remote, CI, release discipline) remain the most important things to do before any more features.

## On the work done

The subagent + adversarial review pattern worked well. Each WI was implemented by a GLM subagent with detailed instructions, then reviewed by a Kimi (K2.7) subagent with a different perspective. The adversarial reviews caught real issues every time:

- **WI-054**: The reviewer caught incorrect STIG control IDs (the initial implementation used fabricated IDs like `WN10-CC-000001`), a wrong CIS control for WDigest, and missing propagation through `estate_doctor` and `report.py`. All were fixed.
- **WI-055**: The reviewer caught missing propagation through `report.py` and CSV export, inaccurate SDDL right codes in remediation text, a wrong cmdlet (`Set-GPPermissions` doesn't change ownership), and inaccurate NTLM level guidance.
- **WI-057**: The reviewer caught that auth failures didn't use the API error envelope, that `run_query` only caught `ValueError`, that the health endpoint leaked the DB basename, and that optional params were silently dropped. A pre-existing `serialize_result` bug with `datetime`/`Enum` types was also found and fixed.
- **WI-056**: The reviewer caught a missing API root catalog entry, an unused DB query in the web route, and a sparkline that silently misbehaved on negative input.
- **WI-060**: The reviewer caught brittle CSS class substring checks (`"gp-table" in html` also matches `gp-table-wrap`), missing `/trends` page coverage, and weak dashboard assertions.

The pattern of "implement with one model lineage, review with another" is genuinely valuable. The reviewer found issues the implementer missed because it approached the code with fresh eyes and a different reasoning style. The cost is ~2x the implementation time, but the quality improvement is significant.

What I'm least confident in: the compliance control IDs. Even after the reviewer's corrections, I can't guarantee every CIS/STIG/NIST mapping is exactly right without cross-referencing the actual benchmark documents. The remediation text is similarly difficult to verify for accuracy — the NTLM level guidance was wrong in the first pass and may still have subtleties.

## Gaps to flag

- **Compliance control ID accuracy is not verified against authoritative sources.** The STIG IDs were corrected by the reviewer but I haven't cross-referenced them against the actual STIG benchmark documents. The CIS and NIST mappings are similarly unverified beyond plausibility. `danger_rules.toml` and `_BUCKET2_COMPLIANCE` in `danger.py` should be reviewed by someone with access to the actual benchmarks.
- **The `serialize_result` fix in `display.py` was a pre-existing bug exposed by the API smoke test.** The function used `dataclasses.asdict()` which doesn't convert `datetime` or `Enum` values, causing `JSONResponse` serialization failures. The fix adds manual recursion with type conversion, but it changes the behavior for all callers (CLI JSON output was previously masking this with `json.dumps(..., default=str)`). This should be tested more thoroughly.
- **The `_print_table` type signature mismatch** (`list[list[str]]` vs `list[Sequence[str]]`) was introduced when the compliance column was added to the danger CLI table. The fix was to cast `[list(r) for r in rows]` at the call site, but a cleaner fix would be to make `_print_table` accept `Sequence[Sequence[str]]` instead.
- **No automated UI tests for POST routes.** WI-060 added 232 GET-based structural assertion tests, but POST routes (ingest, baseline diff, ask, resultant) are not covered. The adversarial reviewer flagged this; it's a known gap.
- **The trend analysis loads the full estate for each snapshot on every page load.** For large estates with many snapshots, this is O(N) heavy work. No caching is implemented. This is acceptable for now but will need optimization if estates grow.
- **The REST API has no rate limiting or caching.** Expensive queries like `danger_findings` and `principal_resultant` can be called repeatedly. The API is read-only so there's no data corruption risk, but a DoS concern exists.
- **Spec files for the new modules don't exist.** `docs/spec/wi_trend.md` was created, but there's no spec for the REST API (WI-057) beyond the AC-07 addition to `wi_query_dispatch.md`, and no spec for the compliance/remediation fields beyond the updates to `wi_danger.md`.
