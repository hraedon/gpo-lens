## On the work done

**WI-030 was the highest-value item.** The Pester test harness is good — 49 tests covering the 8 extracted binding/SNI/cert functions. But the real win was the VM validation, which found two bugs that would have caused silent cert loss on every idempotent installer re-run:
1. PS 5.1 parses bare `[bool]$Sni` as a parameter argument as the string `"[bool]False"`, not as a cast. Fix: `([bool]$Sni)`.
2. The netsh regex `Certificate Hash:` doesn't match real Windows output `Certificate Hash             :` (padded with spaces before the colon). Fix: `Certificate Hash\s*:\s*` + IIS `certificateHash` property as primary source.

The cert preservation bug (#2) was pre-existing — the original script had the same regex. It means every operator who re-ran the installer to upgrade gpo-lens silently lost their TLS cert binding. The script would print "No TLS certificate configured" and the site would serve HTTP 503 until the operator manually rebound the cert. This is a real operational pain point that's now fixed.

The `uninstall-windows.ps1` script was a bonus — it makes the remove/redeploy test cycle trivial and is a useful operational tool.

**The web/app.py split** was clean. GLM handled the tricky parts well: constants that tests patch stay in app.py, route modules look up `_MAX_UPLOAD_BYTES` dynamically at request time, and the `register(app, templates)` pattern is idiomatic FastAPI. The one weakness: the Kimi adversarial review didn't complete fully (it asked for scope clarification instead of just reviewing), but I verified the key concerns directly (architecture boundary, CSRF, audit, security tests all pass).

**WI-042** was a small but correct fix. The `_md_code` function was HTML-escaping content inside Markdown code spans, but CommonMark renderers escape code span content automatically — so `<val>` became `&amp;lt;val&amp;gt;` (double-escaped). The fix removes the HTML escaping and trusts the renderer. This is safe because CommonMark-compliant renderers (GitHub, GitLab, VS Code) escape code span content.

**WI-018** analysis was valuable — the "snapshot/event-store" is really three layers with different cost/value profiles. Layer A (store.py) is foundational and non-negotiable. Layer B (snapshot_diff) is coherent and opt-in. Layer C (events + SIEM sinks) is redundant with Layer B by design and is the genuine simplify candidate. The decision is recorded but not acted on — it's a future consideration.

## Gaps to flag

- **Pester tests can't catch PS 5.1 main-body bugs** — the dot-source guard means the test only exercises the extracted functions, not the main execution body. The `[bool]` cast bug was in the main body's call to `Resolve-EffectiveBinding`. The CI Pester job runs on PS 5.1 but only tests the functions, not the wiring. A future improvement: add an integration test that actually runs the script (with mocked IIS cmdlets) on PS 5.1.
- **`function global:netsh` stub leaks across Pester runs** — when Pester tests and install tests run in the same PowerShell session, the global `netsh` function persists and intercepts real `netsh` calls. This caused a false failure during VM validation (the fresh install used the stub instead of real netsh). The CI Pester job runs in isolation, so this isn't a CI issue — but it's a gotcha for anyone running both in the same session.
- **`S310` ruff ignore is still broad** at `pyproject.toml:54` — `urllib.request.urlopen` in `narration.py` and `sinks.py` could use per-call-site threat model documentation instead of a blanket ignore. Noted in prior reflections, remains.
- **`B007` ruff ignore** at `pyproject.toml:52` — blanket suppression of unused loop variable. Individual `_` fixes would be cleaner. Noted in prior reflections, remains.
- **The cert preservation fix (`certificateHash` fallback)** reads the IIS binding's `certificateHash` property. This property exists on real IIS bindings but NOT on Pester's mocked `pscustomobject` on PS 5.1 — the certHash test is skipped on PS 5.1 with a note. The feature works in production (verified via live VM test) but can't be unit-tested on PS 5.1 due to Mock limitations.
