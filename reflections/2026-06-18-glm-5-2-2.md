## On the project

The IIS deployment story is the live frontier, and it's where the rough edges
are — not in the deterministic core, which is genuinely solid and well-tested.
Three real estate-import failures in a row (upload size, CSRF, install-script
config clobber) were all operational/config-layer, and all stemmed from the
same root: the web app and install script were designed against a loopback-direct
mental model and then bolted onto an IIS+SNI reverse proxy. The codebase already
*half-knew* this (the `_forwarded_proto` middleware, the `deploy/iis/` tree, the
README's SNI section) but the CSRF guard and the install-script binding block
hadn't caught up. That's a coherent, finishable gap — not structural decay.

What's elegant: the ADMX crosswalk (`admx_parser.resolve_display_name`) already
solves the "registry Identity is unreadable" problem and is wired into every CLI
surface — Plan 018 Phase A is mostly a plumbing job to give the web app the same
`--admx-dir` knob the CLI has. What feels fragile: `install-windows.ps1` is ~750
lines of PowerShell 5.1 with a no-BOM/no-non-ASCII constraint that's easy to
violate (I caught an em-dash I'd introduced), and it has no automated tests.
That's the single biggest untested load-bearing artifact in the repo.

## On the work done

Most confident: the CSRF fix. It went through two adversarial reviewers; the
first round surfaced a real empty-netloc bypass and a default-port normalization
gap, both fixed and now tested. The same-host comparison (Origin netloc vs the
request's own Host header) is the correct CSRF signal behind a TLS-terminating
proxy and is robust to whether IIS sets `X-Forwarded-Proto`.

Least confident / the one caveat I want flagged: the `install-windows.ps1` fix
was validated **only** by extracting the detection/effective-value logic and
unit-testing it in isolation with `pwsh` against mock bindings. The full IIS path
(`Get-WebBinding` / `netsh` / `New-WebBinding` / `Set-ItemProperty bindings`)
and the three idempotency skip-branches are unexercised end-to-end. I traced
five scenarios by hand (upgrade no-op, port override, cert rotation, add-SNI,
fresh install) and they hold, but "traced by hand" ≠ "tested on Windows." That's
WI-030. I did not run a Windows smoke test because there's no Windows host in
this environment.

Awkward: the PowerShell no-BOM/ASCII constraint. The script header documents it,
but it's a footgun — an em-dash inside a `Write-Host` double-quoted string is
exactly the parse-corruption pattern the header warns about, and I introduced
one and only caught it on review. A CI lint that rejects non-ASCII in `.ps1`
files would be cheap insurance.

  with the other session (and a choice between the two Plan-018 Phase-B framings)
  before they're committed. Treat the working tree as shared/in-flight.
- **`install-windows.ps1` is untested on Windows** (WI-030). This session's
  config-preservation fix is logic-validated only. The SNI/binding/cert
  mutation paths could regress silently.
- **CSRF deferred items** (from adversarial review, intentionally not fixed):
  `requestTimeout="00:04:00"` is fine for IIS→uvicorn loopback but could abort
  large uploads over a slow browser→IIS link; the `!= 403` test assertion pattern
  is intentional (decouples from route response codes) but weaker than asserting
  a specific status. Low value, noted not filed.
- **No CI lint for non-ASCII in `.ps1`.** The no-BOM constraint is documented
  in the script header but not enforced. One em-dash in a double-quoted string
  would have corrupted PS 5.1 parsing on a GitHub-zip download.
- **`X-Forwarded-Proto` assumed present.** The `_forwarded_proto` middleware
  assumes IIS sends it; the new CSRF same-host check deliberately does *not*
  depend on it (host:port only), so this is latent rather than blocking — but
  if IIS isn't sending it, `url_for` emits `http://` links (mixed content) and
  nobody has verified IIS actually sets it in production.
