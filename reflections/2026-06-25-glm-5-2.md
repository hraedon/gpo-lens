## On the project

The architecture is the project's strongest asset. The deterministic/AI separation, import-boundary enforcement via architecture tests, stdlib-only Slice 1, and "flag don't simulate" topology stance are all disciplined design decisions that most security tools lack. The coverage-gap reconciliation (comparing gpo-inventory.json against collection-errors.json) shows real-world maturity — it doesn't pretend to see what it can't see.

What's most striking is the gap between the strength of the core and the thinness of the feature layer. The data model, parsers, merge engine, snapshot diffing, danger detection, and topology resolution are all genuinely well-built. But the tool stops at "here's what's wrong" — it doesn't map to compliance frameworks, doesn't tell you how to fix things, doesn't visualize trends, and doesn't expose an API for integration. The roadmap filed this session (WI-054 through WI-062) addresses these gaps. The natural sequence is: compliance mapping + remediation (schema extensions, no new infra) → trend analysis (existing SQLite data) → REST API (thin layer over query_dispatch.py) → the lower-severity items.

The spec docs have drifted from the code in several places — not just the constant rename (which I fixed), but phantom private helpers listed in wi_topology.md that haven't existed since the WI-046/WI-047 consolidation, and 9 missing exports in wi_authz.md's public surface table. The project would benefit from a spec-to-code drift check, similar to how it already has architecture tests for import boundaries.

## On the work done

The WI-052 rename was straightforward — three files, one constant, no behavioral change. The ruff import-sort fix was the only complication (the new name `READ_OR_APPLY_RIGHTS` starts with R, not A, so it moved in the sort order). The explanatory comment I added to the constant definition explicitly calls out *why* the name is `READ_OR_APPLY_RIGHTS` and not `APPLY_RIGHTS`: GR and RP are read-only rights, GA includes write. That comment is the real fix — the rename alone wouldn't prevent someone from re-introducing the misleading name.

The WI-053 CSS fix was one line: `overflow: hidden` → `overflow-x: auto`. The prior session's reflection correctly diagnosed the root cause but didn't fix it (the user wanted to retest first). A prior commit (`c4050d7`) had added `.mono { overflow-wrap: anywhere; word-break: break-word; }` which helps break long words within cells, but the `overflow: hidden` on the container was still clipping entire columns when the table exceeded the viewport width. My fix removes the redundant media query rule since `overflow-x: auto` is now the default on all screen sizes.

The spec doc updates were the most satisfying cleanup. Finding three phantom private helpers in wi_topology.md (`_SDDL_READ_OR_APPLY_RIGHTS`, `_grants_read_or_apply`, `_broad_key`) that haven't existed since the WI-046/WI-047 consolidation — and removing them — is exactly the kind of doc drift that wastes the next agent's time. The 9 missing exports in wi_authz.md's public surface table were similarly stale.

The identifier-gate failure was not my fault (the prior session leaked `WORK-DOMAIN.local` in a reflection file) but I fixed it because it was blocking CI. This is the second time a reflection file has leaked a work-domain identifier. The project should consider adding `reflections/` to the identifier gate's scan path, or at minimum, agents should be more careful when writing reflections that reference sample estate names.

## Gaps to flag

- **`docs/spec/` drift is systemic, not just the constant rename.** The spec files reference private helpers that were deleted during consolidation, and the public surface table in wi_authz.md was missing 9 exports. A spec-to-code consistency check (similar to the architecture import-boundary test) would catch this automatically.
- **`reflections/` directory is not scanned by the identifier gate.** The `WORK-DOMAIN.local` leak went undetected until CI ran. The `check_committed_identifiers.py` script should scan all committed files including reflections and docs.
- **`detection.py:761`** — the `_is_default_writer_sid = is_default_writer_sid` alias is still present. Unlike `_permission_implies_read` (which I removed), this one IS imported by `tests/test_authz.py`, so it's a legitimate backward-compat shim. But it's the only remaining private alias of a consolidated function, and it creates confusion about where the canonical implementation lives.
- **The `SddlApplyAce` dataclass name** (`authz.py:427`) still uses "Apply" in its name, even though the rights set is now `READ_OR_APPLY_RIGHTS`. The class name is fine because it represents "an ACE that grants apply" (which includes read-implying rights), but it's worth noting the naming asymmetry if someone later questions why the dataclass says "Apply" but the constant says "READ_OR_APPLY."
- **No automated UI tests exist.** WI-060 is filed but not implemented. Every web template change is verified manually. The reflection from the prior session flagged this, and I'm flagging it again because it's the most likely source of silent regression.
