## On the project

gpo-lens is a well-engineered security analysis tool with strong fundamentals. The codebase demonstrates genuine security awareness: defusedxml, timing-safe token comparison, four-layer zip extraction defense, parameterized SQL everywhere, CSV injection mitigation, comprehensive CSRF testing. The import boundary (no AI in the deterministic core) is enforced by an AST-based architecture test. The fixture quality is high — cpassword, MS16-072, version skew, loopback, ILT, coverage gaps are all covered.

What feels fragile: the SDDL parser is hand-rolled and handles the common cases well but misses edge cases (conditional ACEs, hex rights masks) that a library like `pysspilib` or `python-ad` would handle natively. The `canonical_guid` hyphen bug was a latent landmine — it worked because the collector always emits hyphenated GUIDs, but any non-standard input would silently break joins. The async web routes making blocking SQLite calls is a known architectural choice (WAL mode helps but doesn't fix the event-loop blocking).

The breadcrumb/work-item lifecycle has a UX issue: `agent-notes breadcrumb update --status in_progress` maps to `claimed` (a liveness axis) instead of the lifecycle state `in_progress`, making it impossible to transition work items from `open` through the canonical lifecycle from the CLI. This should be filed against agent-notes, not gpo-lens.

## On the work done

The parallel-agent methodology worked well for this codebase. The 4-reviewer split by attack surface (web, core, logic, tests) produced minimal overlap and good depth. The 6-fixer split by file ownership avoided merge conflicts entirely — the only integration issue was an unused import in test_merge.py.

What went well:
- Every critical/high finding was verified by running actual code before synthesis
- The canonical_guid fix correctly cascaded through all test fixtures (7 files updated)
- The adversarial reviewers caught real issues the fixers missed (OverflowError, blocked-skip in absent loop, case-sensitive CSE comparison in two additional modules)

What I'd want a second pair of eyes on:
- The M-4 fix (overbroad_apply_group_policy) rewrote a large function. The `broad_trustee_key` call for SDDL aliases (`wd`, `an`) doesn't canonicalize to SID form, so an allow for `wd` and a deny for `S-1-1-0` won't cancel in `applies_broadly`. Filed as WI-084.
- The M-9 fix (permission_implies_apply in security gate) changes merge behavior. Tests pass, but the principal resultant is complex enough that edge cases in security-filtered estates deserve manual verification against a real export.
- The `attach_sysvol_paths` fix (rebuilding hyphenated GUIDs from bare canonical form) adds three candidate paths per GPO. On large estates this is O(n) filesystem stat calls — fine for 100 GPOs but worth noting.

## Gaps to flag

- `store.py:_iso_to_dt` — now wraps ValueError (fixed this session), but the broader pattern of unguarded `datetime.fromisoformat` should be audited project-wide
- `detection.py:641` and `_admx_coverage.py:93,141` — had the same case-sensitive CSE bug as danger.py; fixed this session, but the pattern should be centralized (e.g., a `is_registry_cse(s)` helper)
- Test fixtures in `test_danger.py`, `test_scope_honesty.py` etc. still construct `Gpo(id="11111111-2222-...")` with hyphens — non-canonical but doesn't break tests. Could mask join bugs if a test ever crosses canonical vs hand-built IDs.
- The `agent-notes` CLI lifecycle transition bug (`in_progress` → `claimed` instead of lifecycle state) means work items fixed in code can't be properly transitioned through the review gate from the CLI. This is an agent-notes issue, not a gpo-lens issue.
- The `_HEX_RIGHTS_MAP` in authz.py maps ADS_RIGHTS_ENUM bits to SDDL codes, but `SYNCHRONIZE` (0x00100000) has no SDDL code and is silently skipped. This is correct behavior (SYNCHRONIZE is not a read/apply/write right in the SDDL model) but should be documented.
