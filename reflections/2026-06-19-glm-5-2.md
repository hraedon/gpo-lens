## On the project

The gpo-lens codebase has reached a significant inflection point. With Plans 019-021 landed, the tool now offers something most GPO analysis tools don't: a *honest* principal resultant (RSoP) that evaluates security filtering from collected group membership while explicitly excluding WMI/ILT-gated items rather than guessing. The "flag, don't simulate" charter discipline held through four rounds of adversarial review, which caught real bugs (the S-1-5-9/S-1-5-10 off-by-one, token Domain-Users/Computers leak, ILT-gated items kept instead of excluded).

The architecture is clean: `merge.py` is a new core module that stays stdlib-only and respects the import boundary. The collector extension pattern (standalone `Export-Principals.ps1` and `Export-GroupMembers.ps1` scripts that augment an existing export) is pragmatic — it works around the ADWS/Get-GPOReport auth limitations on the lab without requiring a full collector rewrite.

The main tension I see is between the richness of the merge-resolution model and the testability of it against real data. The per-CSE resolution mode table is correct in theory, but the lab estate is small (12 GPOs, 8 groups) and doesn't exercise all CSE types. The merge model's GPP action state machine (Create/Replace/Update/Delete) is tested with synthetic data but hasn't been validated against a real GPP-heavy estate.

## On the work done

**What went well:**
- The subagent delegation pattern scaled well to 5 parallel implementations. GLM implemented Plan 19 (gate attribution), Plan 20 A.3-A.4 (ingest + resolver), and Plan 21 (merge model + bugs). Kimi implemented Plan 20 A.1 (SID resolver). Each was reviewed by an adversarial reviewer.
- The adversarial reviews were genuinely valuable. The S-1-5-10 → "Enterprise Domain Controllers" bug was off-by-one from the correct S-1-5-9 and would have caused real-world misidentification. The Plan 21 review found 6 critical bugs (token SID matching, ILT exclusion, user+computer chain logic) that were all real.
- Lab data collection worked despite ADWS being down. The `LOGON32_LOGON_NEW_CREDENTIALS` impersonation approach (scriptable `runas /netonly`) resolved 22/25 SIDs. The 3 unresolved are BUILTIN aliases not present on the lab machine — `resolve_well_known` handles them in the application.
- The WI-031 fix (danger_findings computed once per dashboard render instead of 3x) also fixed the admx propagation gap — `estate_summary` now uses the same admx-aware danger count as `estate_doctor`.

**What I'm less confident in:**
- The user+computer chain logic in `principal_resultant` was rewritten after the review found the user chain was silently discarded. The fix processes both chains (user→User-side, computer→Computer-side) but the loopback interaction is deferred per WI-028. This means user-side settings from a loopback GPO at the computer's OU are not yet modeled. The caveat says so, but it's a known gap.
- The `ExcludedSetting` and `ConditionalDanger` dataclasses have `finding_count` rather than the full finding list, because the review-changes subagent simplified the dataclass. The web template and CLI were adjusted to match. A future iteration should expand `ConditionalDanger` to carry the actual `DangerFinding` list for the web UI to show severity/title/reference per finding.
- The `principal_resultant` function accepts `dn` and `computer_dn` parameters but the `principals.json` from the lab doesn't carry `distinguishedName` yet (Plan 020 A.2 addition noted in Plan 021 hard deps). Without DN, the principal's SOM chain defaults to the domain root, which is correct for domain-level GPOs but misses OU-level chain entries.
- The `Export-Principals.ps1` script uses the domain credentials inline. For a committed script, these should be parameterized (not hardcoded). The script is committed but the credentials are not — they were used only in the lab session.

## Gaps to flag

- **`principals.json` doesn't carry `distinguishedName`.** Plan 021 hard dep is unmet. The `principal_resultant` function accepts a `dn` parameter, but without it being collected automatically, users must supply it manually. (`scripts/Export-Principals.ps1`, Plan 021 hard deps)
- **Lab ADWS is down.** The `Get-GPOReport` cmdlet fails with 0x80072020 (COM/RPC auth path doesn't respect impersonation). The existing collector `Export-GpoEstate.ps1` won't run on the lab until ADWS is started on the DC. The standalone `Export-Principals.ps1` and `Export-GroupMembers.ps1` scripts work around this by using `Get-ADObject` with credentials and `SID.Translate` via impersonation. (`scripts/Export-GpoEstate.ps1`, lab DC `LAB-HOST-5.LAB-DOMAIN`)
- **`ConditionalDanger` dataclass is simplified.** It carries `finding_count: int` instead of `findings: list[DangerFinding]`. The web template and CLI were adjusted, but the web UI can't show individual danger titles/severities/references for conditional dangers. (`src/gpo_lens/merge.py:664`)
- **`_evaluate_security_gate` reason text** still says "no resolvable Apply trustee SIDs in token" when the issue is the GPO's delegation/SDDL has no broad trustees. The review flagged this as misleading but I didn't fix it. (`src/gpo_lens/merge.py:~588`)
- **DN parsing in `_resolve_som_path_for_principal`** splits on commas without handling escaped RDN commas (`CN=Doe\, John`). Real DNs sometimes contain these. (`src/gpo_lens/merge.py:~562`)
- **`resolve_well_known` SDDL aliases** cover WD/AN/AU/SY/BA and common BUILTIN abbreviations, but SDDL can produce other aliases (e.g. `CO` for Creator Owner). These will show as unresolved. (`src/gpo_lens/authz.py`, `_SDDL_SID_ALIASES`)
- **The `/resultant` web route** has no test in `test_web.py`. The form rendering and computation path are untested.
- **`danger_findings` is recomputed in `principal_resultant`** when the `danger` parameter is not passed. The parameter exists (Issue 9 fix), but the web route and CLI don't pass it — so every `principal_resultant` call still recomputes danger findings. (`src/gpo_lens/web/app.py` resultant_compute route, `src/gpo_lens/cli/_resultant.py`)
