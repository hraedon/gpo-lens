## On the project

The project's charter discipline continues to impress. "Flag, don't simulate"
isn't just written in AGENTS.md — it's faithfully executed in every scope-caveat
surface, and the two caveat formats (OU-level `"loopback=<mode>"` vs GPO-level
`"Loopback mode: <mode>"`) are a natural consequence of two different query
functions serving two different views. The architecture boundary test keeps
the deterministic core AI-free. The security posture (CSRF, CSP, zip extraction
hardening, constant-time token comparison, read-only SQLite) is genuinely
strong for a local-first tool.

The tension I noted last session — between the "air-gappable stdlib-only core"
and the FastAPI web UI — is now fully resolved operationally. The `-Sni` and
`-WindowsAuth` switches make the IIS deployment story production-ready: gpo-lens
can share port 443 with cert-watch via SNI, and access control is a one-flag
affair. The first Windows smoke test proved the install script works end-to-end,
which was the single highest-risk gap. The "untested on Windows" caveat that's
been in every recent reflection is now retired.

## On the work done

**The smoke test was the most valuable thing this session.** Not because it
validated that the code works (the tests do that), but because it caught a
real-world configuration gap that no unit test could: the Windows Auth role
service wasn't installed on the server, and anonymous auth was enabled. The
site would have been **open in production** — anyone with network access would
have been treated as the trusted local analyst. This is exactly the class of
bug that only a real deployment surfaces. I installed the role service, enabled
Windows Auth, verified 401 for unauthenticated and 200 for authenticated
requests, and added the `-WindowsAuth` switch so future deployments don't
repeat the gap.

**The CI identifier gate proved its value immediately.** On the first CI run
with the secret configured, it caught the real work-domain DNS name in
`plan-015`'s author line — a real leak from a prior session. That's the gate
working as designed.
But the GPO-name extraction was a false start: extracting all 735 GPO names
produced **19,987 false positives** because common words like `time`, `load`,
`registry` (which happen to be GPO names in the estate) match everywhere in
`uv.lock`. I fell back to domain-name-only identifiers (2 entries), which is
the right trade-off: domain names are the highest-value identifiers, long
enough to avoid substring false positives, and directly reveal the org. GPO-
name coverage could be restored later with word-boundary matching or a common-
word filter, but the domain names are the 80/20.

**The adversarial review process caught real issues.** Round 1 found a UTF-16
bypass in the identifier gate (null bytes in UTF-16-LE were misclassified as
binary), an OOM risk (reading entire files for the binary sniff), a vacuous
test (the binary-skip test didn't put the identifier in the binary file), and
an audit-logger race condition. Round 2 found a silent `Install-WindowsFeature`
failure path and a late SNI validation that modified the site before throwing.
All were fixed before the final push. The player-coach split (delegate
implementation to kimi/glm, review with adversarial subagents) continues to
work well.

**PowerShell-through-SSH was the worst part of the session.** The quoting is
a layered nightmare: local bash → SSH → remote PowerShell 5.1, where `$`
variables are consumed, `echo` is an alias for `Write-Output` (which requires
`InputObject`), `2>NUL` is CMD syntax that PowerShell rejects, `<` and `>` are
reserved operators, and em-dashes in scripts corrupt the ANSI codepage parser
(documented in `install-windows.ps1`'s header but easy to forget). The only
reliable approach was writing `.ps1` files locally, `scp`-ing them to the
server, and executing with `powershell -ExecutionPolicy Bypass -File`. I should
have started with that pattern instead of fighting inline quoting for 4
iterations.

**What I'm confident in:** the install script changes (SNI ordering, WindowsAuth
role-service handling, early validation), the identifier gate (UTF-16 + streaming
+ binary-skip test all fixed), the audit logging (race lock + filename cap), and
the backup/restore runbook. All verified on a real Windows host.

**What I'd want a second pair of eyes on:** the SNI code path. The smoke test
used 8443 non-SNI (per the user's call). The SNI path handles the error-87
ordering correctly (sslFlags before hostnameport sslcert) and the adversarial
reviewer verified the IIS/http.sys layer separation, but it hasn't been
exercised on a real SNI deployment sharing port 443. It's code-reviewed and
logically correct, but "logically correct" for IIS TLS binding is not the same
as "is correct" — that class of Windows configuration has too many undocumented
interactions.

## On what remains

  The smoke test used 8443 non-SNI. To validate: deploy with
  `-Sni -Port 443 -HostName gpo-lens.WORK-DOMAIN.local` and verify both gpo-lens
  and cert-watch respond on their respective hostnames.

  Splunk holds (AD event 5136 / SYSVOL 4663) before any code. That's a

## Gaps to flag

- **`scripts/install-windows.ps1:496-522` — SNI path untested on Windows.**
  The code handles error-87 ordering (sslFlags before hostnameport) but was
  never exercised on a real SNI deployment. The prior error-87 failure
  (cert-watch WI-047) is documented in the replaced comment block; the new
  code's ordering is correct by analysis but not by execution.
- **CI secret contains only 2 identifiers (domain DNS names).** The extraction
  script (`/tmp/extract_ci_identifiers.py` — not committed, ephemeral) tried
  735 GPO names and produced 19,987 false positives. Domain-name-only is the
  pragmatic choice but leaves GPO-name leaks uncovered. A future iteration
  with word-boundary matching could close this.
- **LAB-HOST-1 deployment is manually configured.** The Windows Auth and TLS
  binding were configured via direct PowerShell commands during the smoke test,
  not via the `-WindowsAuth` switch (which was added after the manual config).
  A clean redeploy with the switch would validate the automation end-to-end.
- **`scripts/check_committed_identifiers.py` uses substring matching.** This
  is by design (real identifiers can appear as substrings of longer tokens),
  but it means short identifiers cause disproportionate false positives. The
  4-char minimum helps but isn't enough for common technical terms. The
  domain-name-only secret sidesteps this; broader identifier sets would need
  word-boundary or frequency-based filtering.
- **No breadcrumb filed for the GPO-name false-positive finding.** The CI
  gate's identifier set is domain-name-only because GPO names were too noisy.
  This trade-off is documented in commit `ef8fd0e` but not in a breadcrumb.
  If the next agent wonders why the gate doesn't catch GPO-name leaks, the
  answer is in the commit log, not in an active work item.
