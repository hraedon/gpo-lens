## On the work done

**What went well:** The SDDL Read/Apply conflation fix was the highest-value
correctness fix. The `READ_OR_APPLY_RIGHTS` set included `GR` (Generic
Read) and `RP` (Read Property), which meant a GPO with only Read to
Everyone triggered the overbroad-apply danger finding (false positive) and
passed the security gate in merge.py (over-grant). The fix — a separate
`APPLY_RIGHTS = {"GA", "CR"}` — is validated against real GPMC SDDL from
the reference corpus: the Apply Group Policy ACE is an Object ACE with
`CR` rights and GUID `edacfd8f-ffb3-11d1-b41d-00a0c968f939`. The
asymmetry (deny check uses `READ_OR_APPLY_RIGHTS`, allow check uses
`APPLY_RIGHTS`) is correct Windows behavior: denying Read blocks Apply,
but granting Read doesn't confer Apply.

**What I'm confident in:** The hash chain is correct for its threat model
(single-writer, single-process, local SQLite). The `verify_event_chain`
gracefully handles pre-migration events (NULL prev_hash) and missing
columns. The rate limiter is a reasonable backstop for the deployed
single-worker uvicorn — the IIS reverse proxy is the real defense.

**What I'd want a second pair of eyes on:** The `ingest.py:625` decision
to keep `cse.strip().lower() == "registry"` instead of `is_registry_cse(cse)`.
The WI-2 subagent decided this is intentionally narrow — only the "Registry"
CSE (not "Windows Registry") has `<Policy>` blocks for Administrative
Templates. I agree with the reasoning, but if GPP Registry preferences ever
emit Policy blocks under the "Windows Registry" CSE, they'd be routed to
the wrong parser. In practice this doesn't happen, but it's a latent edge
case.

The WI-5 GUID sweep touched 157 literals across 30 test files. I verified
the key test files pass, but a 157-literal sweep is the kind of change
that could have a subtle breakage in a file I didn't run. CI is green, so
this is likely fine, but the scope is large.

## Gaps to flag

- **`merge.py:631-655`** — The SDDL fallback security gate doesn't expand
  `GA` into its component rights before checking deny. A deny of `GR`
  doesn't remove `GA` from the allow set (they're different 2-letter
  codes), so `net = allowed - denied` still contains `GA`. The GPO
  applies despite the read deny. This is a pre-existing limitation of
  the 2-letter-code-level SDDL parser, not introduced by this session's
  changes. A proper fix would expand `GA` → {GR, GW, GX, RC, ...} before
  the deny subtraction.
- **`events.py:87`** — `append_events` uses a single timestamp for the
  entire batch. All events in a batch get the same timestamp, losing
  sub-second ordering. Not a bug (events are ordered by id), but could
  confuse consumers that sort by timestamp.
- **`web/rate_limit.py`** — In-memory state is per-process. If multiple
  uvicorn workers are ever used, each gets its own counters and the
  effective limit is multiplied. Documented in the module docstring but
  not enforced.
- **`test_cli_estate_coverage.py`** — The `test_show_text_with_known_gpo`
  test passes the canonical GUID on the command line. If the `show`
  command ever changes to require a different ID format, this test would
  break silently (it only checks return code and output presence, not
  specific GPO content).
- **Regista writes are now unblocked** — the migration that was blocking
  all agent-notes operations has been applied. The PENDING-REGISTA-WI.md
  holding pen can be deleted.
