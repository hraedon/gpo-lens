## On the project

gpo-lens is well-architected for its constraints: read-only, deterministic, no AI in the truth path. The data model (`model.py`) is clean and the separation between ingest, detection, queries, and topology is well-judged. The codebase's main weakness is that the same conceptual operations — "extract apply/read trustees from a GPO", "check if a permission implies read", "look up a GPO by id" — get reimplemented in each module that needs them, with subtle variations that drift. The `authz.py` module was a good start at centralizing SDDL/trustee logic, but it hasn't been pushed far enough — the delegation-first-then-SDDL fallback pattern (WI-046) and the "default writer" / "permission implies read" predicates (WI-047) are still duplicated 2-4x each.

The `Setting.raw: dict[str, object]` field (WI-048) is the root cause of a lot of defensive complexity. Every consumer does `isinstance(raw, dict)` then gropes through `raw.get("children")`, `raw.get("@attr")` etc. The `_extract_loopback_mode` function in topology.py is 75 lines handling three raw-dict shapes — that's a direct consequence of the untyped raw. A TypedDict for the `element_to_dict` output would make the structure an invariant rather than something every consumer must probe.

## On the work done

The fixes were low-risk and mechanical, but the Kimi review earned its keep. It caught that `_load_json_records` silently returned `[]` for non-list/non-dict top-level JSON (a coverage-honesty charter violation), which I fixed. It also noted that the `_child_by_localname` wrappers in ingest.py and detection.py were "renamed, not removed" — I then eliminated them via aliased imports (`from ... import child_by_localname as _child_by_localname`), which avoided touching 83+ call sites.

I'm confident in the `APPLY_RIGHTS` consolidation and `_broad_key` deletion — those are pure mechanical dedup with no behavioral change. The `_load_json_records` helper is the one I'd want a second pair of eyes on: the warning message changed from per-entry ("Skipping non-dict entry") to batched ("Skipped 3 non-dict entries"), and the `stacklevel` changed from 1 to 2. Tests were updated but the warning semantics shift is a behavioral change that could affect downstream consumers parsing warnings.

The `_try_load` / `_try_action` helpers use `or []` / `or {}` to default `None` returns. The Kimi reviewer correctly noted this is fragile for falsy returns, but all current callers return containers where this is safe. I left it as-is for readability; a future `is None` check would be more defensive.

## On what remains

1. **WI-045 (high)** — Estate GPO-by-id index. This is the highest-impact refactor: it eliminates O(n) lookups, removes 5+ duplicated `{g.id: g}` dicts, and fixes the O(n²) scan in `excessive_writers`. Should be done first as it's a prerequisite for clean topology/merge code.
2. **WI-046 (medium)** — Delegation+SDDL fallback consolidation. Depends on having a clean Gpo method or authz function for extracting apply trustees. Can be done independently but benefits from WI-045.
3. **WI-047 (medium)** — Duplicated authorization predicates. Quick to fix once the functions are in authz.py.
4. **WI-048 (medium)** — Setting.raw typed. This is the biggest structural change and would touch the most code. Best done as a dedicated sprint with its own test plan.
5. **WI-049/050/051 (low)** — Defensive improvements that are individually small but improve the code's robustness.
