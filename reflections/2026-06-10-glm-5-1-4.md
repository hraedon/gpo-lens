# Session Reflection — 2026-06-10

**Work summary:** Implemented Plan 010 Workstream C (Tier 3: LLM narration) — design doc, `doctor --explain`, `ask` NL query routing, adversarial reviews with bug fixes. Also decomposed `queries.py` into `queries.py` + `detection.py` and refreshed the README. 280 → 326 tests. All gates green.

## On the project

gpo-lens is hitting its stride as a project. The architecture is clean enough that a 46-test, two-module addition (narration + routing) landed without any cross-module pain. The re-export pattern in the decomposition means zero test changes were needed despite moving 10 functions and 3 dataclasses to a new module. That's a sign the original module boundaries were thought through.

The weak spot is still `cli.py` at 1579 lines. The `_QUERY_DISPATCH` dict inside `cmd_ask` is a 40-line block that re-lists every query function. Adding a new query now requires editing three places: `detection.py` (the function), `queries.py` (re-export), `cli.py` (dispatch table + maybe a dedicated command), and `narration.py` (`_VALID_QUERIES` + routing prompt). That's four files for one new query. A registry pattern would help, but I didn't introduce it this session because it would touch too many files for too little immediate value.

The narration layer is deliberately thin — `urllib.request`, no deps, 159 lines. It's enough to prove the seam works. The real value will come when the prompts are tuned against real doctor findings from a messy estate, which hasn't happened yet (all testing is against synthetic fixtures).

## On the work done

**What I'm confident in:**

- The adversarial review pipeline caught real, dangerous bugs in both rounds. The WI-C.2 review found that the CLI integration tests were dead code (pytest never collected `DoctorExplainTests`) and that subprocess mocks don't cross process boundaries. The WI-C.3 review found cpassword exfiltration to the LLM and dataclass serialization producing strings instead of JSON objects. Both were serious. Both were fixed.
- The decomposition of `queries.py` is clean. 1932 → 1387 lines. Detection scanners in one place, compositional queries in another. Re-exports preserve backward compatibility.
- The README is now honest — collector → ingest → doctor in three commands, which is what a new user actually needs.

**What I'm less confident in:**

- The `ask` command's routing corpus tests only validate the JSON parser, not the LLM's ability to route questions. The test mocks `call_llm` to return the expected JSON, then asserts the parser handles it. A real routing failure would pass these tests. This is documented honestly in the test file, but it means AC-C3-01 is only partially satisfied.
- The narration prompts haven't been tested against real, messy estate data. The synthetic fixture has 8 GPOs; a real work domain has 129. The prompts may need tuning for volume, and the 2048-token `max_tokens` limit may be too low for large estates.
- The `cmd_ask` narration path sends serialized query results to the LLM. For `estate_doctor` on a large estate, that's potentially thousands of findings in one prompt. No truncation or summarization happens before the LLM call.

**What went well:**

- The "player coach" model — coding agents implement, adversarial reviewers catch bugs, I coordinate and fix — worked efficiently. Two rounds of review, 11 real bugs caught, all fixed within the session.
- The decomposition was painless thanks to re-exports. Zero test changes needed.
- The design doc (WI-C.1) written upfront made the coding agents' work much more focused. The ACs gave the reviewers concrete criteria to check against.

**What was awkward:**

- The coding agent for WI-C.2 initially produced tests that never ran (wrong class name). This is the same class of failure the previous reflection flagged — agents reporting success without verification. I caught it via the adversarial reviewer, but it's a persistent pattern across sessions and models.
- `ruff` line-length violations keep showing up in agent-generated code. The agents know the 100-char limit but don't always respect it in practice.

## On what remains

2. **Provider-bench evaluation** — the `ask` routing and `doctor --explain` prompts need testing against real data and potentially multiple models. The `provider-bench` project is right there for this.
3. **Prompt tuning** — run `doctor --explain` against the WORK-DOMAIN.local work estate (129 GPOs) and evaluate the narration quality. Adjust max_tokens, prompt structure, and truncation strategy accordingly.

Less obvious:

- **`ask` integration test** — gated on `GPO_LENS_API_KEY`, sends real questions to a real LLM and asserts the routing response contains a valid query name. This is the missing piece for AC-C3-01.
- **Breadcrumbs cleanup** — `changelog-over-time.md` and `delegation-audit-deep-dive.md` are partially resolved. Decide whether to close or keep.
- **Generator-driven fixture builder** — the hand-crafted XML fixture is a maintenance time bomb flagged by two reflections now.

## Gaps to flag

- `test_routing_corpus.py` tests the parser, not the LLM's routing ability. The test name and docstring are honest about this, but AC-C3-01 is only partially satisfied. A live-LLM integration test gated on `GPO_LENS_API_KEY` would close the gap.
- `cmd_ask` has no token budget management — large estates could exceed the LLM's context window or `max_tokens`. No truncation, no summarization, no error message suggesting `--no-narrate`.
- `_QUERY_DISPATCH` in `cli.py:1113-1151` and `_VALID_QUERIES` in `narration.py:100-116` must be kept in sync manually. No test verifies they match.
- The `ask` subcommand requires an API key even for routing, but the routing prompt is ~500 tokens and the response is ~20 tokens. A local routing model or cached routing table could eliminate this dependency for common questions.
- `detection.py` imports `re` and `xml.etree.ElementTree` at module level, even though only `broken_refs` and `cpassword_scan` use them. Not a bug, but lazy module loading would reduce import time for callers that only need `unlinked_gpos` (a one-liner).
- The `report --out report.html --format html` command in the README hasn't been manually verified in a browser. The HTML generation exists in `report.py` but nobody has checked `@media print` CSS or dark mode.
- `reflections/` directory has 10 files now. No index or cross-session summary exists. An agent starting a new session reads only the latest reflection via `/start`, which means lessons from earlier reflections may be lost.
