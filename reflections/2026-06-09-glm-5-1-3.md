The store schema (SQLite with snapshot_id foreign keys) was clearly designed with this feature in mind — the `setting` table already had `(snapshot_id, gpo_id, side, cse, identity)` as the natural join key. Implementation was ~80 lines for the query, ~45 for the CLI handler.

## On the work done

Straightforward session. The only decision point was choosing which breadcrumb to tackle. Per-setting delta was the clear winner: highest value (AGPM replacement), no blockers, store already supports it.

The `snapshot_settings_diff` implementation uses a dict-join pattern rather than a full SQL JOIN — load old and new settings into dicts keyed by `(gpo_id, side, cse, identity)`, then diff the union of keys. This is O(n) in the total settings count and avoids complex SQL. For the real work domain (1,000+ SOMs, ~5K settings) this is fine. For estates with millions of settings, a streaming approach or SQL-side join would be needed.

One thing I'm confident about: the filter parameters (`gpo_id`, `side`, `cse`) are pushed into the SQL WHERE clause, not applied post-hoc. This means filtered queries only load the relevant rows.

## Gaps to flag

- `snapshot_settings_diff` doesn't paginate — for very large estates, the full result set is materialized in memory. Not a problem at current scale but worth noting.
- The `diff-settings` CLI takes snapshot IDs by number — the user needs to run `snapshots` first to find them. A `--latest` flag comparing the two most recent snapshots would be more ergonomic.
