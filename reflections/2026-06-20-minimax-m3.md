## On the project

The project continues to set a high bar. The architectural boundary (core never imports narration/web) is the right constraint and the AST-based enforcement in `tests/_arch.py` makes it impossible to violate accidentally. The plan-driven development pattern (Plans 007–021) with a `wi_*.md` spec for each module is paying off — `merge.py` at 982 lines is *legible* because every behavior is traceable to a plan decision, and the new `wi_merge.md` codifies that traceability for the most complex post-Tier-1 module.

The two open architecture-debt breadcrumbs (queries.py god module, web/app.py monolith) remain the project's main growth pain. Both are real but neither is urgent — the code is stable and tested, the cost of leaving them is cognitive load rather than bug risk.

## On the work done

The two-round adversarial review pattern caught real bugs and spec drift:
- **First review of WI-034** found dual `--cov-fail-under` source-of-truth between `addopts` and CI; I fixed it by removing the redundant CI flags.
- **First review of TOML fix** found that the initial fix only handled the missing-keys category, not non-dict entries or non-list `rules` — both would have crashed the parser. Critical: my fix would have *passed tests* if I'd added tests for only the missing-keys case. Adversarial review caught the actual crash modes.
- **First review of wi_merge.md** found 3 critical + 5 significant spec/code mismatches. The second review then found 4 more (different reviewer, fresh read). 12 total findings before commit.

The pattern is clear: **adversarial review on a small change pays for itself when the change has spec-like or trust-boundary implications.** For mechanical CI workflow edits, it's overkill. For TOML parsing and spec docs (both are *contracts*), it's necessary.

The single most useful moment: the second review caught that AC-04's "same-order ties included" wording was wrong (the implementation uses `!=`, not `<` or `>`). I'd added that wording in the first round of fixes without re-reading the code. The review forced me to re-read the actual `if/else` branches and discover the asymmetry I had invented.

The TOML fix was a 5-line change in the code but a 5-test change in tests, and the *tests* are what make the fix safe. The tests assert both the skip behavior AND the warning content — so a future change that silently drops the warning or changes the message format will break a test.

## Gaps to flag

- **Spec/code drift is the dominant risk in `merge.py`.** The two-round review found 12 mismatches between the spec I was *writing* and the code that *already existed*. This is a feature, not a bug — it surfaced drift the prior sessions left behind. But it also means: any new `merge.py` PR is a 2-round-review project by default. The bar should be documented somewhere.
- **`overridden_by` ordering is a known simplification** (AC-16). I confirmed it matches the implementation, but the field semantics are subtle (ascending chain order, not sorted descending). A future caller's natural assumption will be wrong; consider sorting at construction or renaming the field to `overridden_by_chain_order`.
- **`MergedSetting.conditional` is reserved/always-False** (AC-19). The field exists, the caveat summary counts it, no code sets it True. Either it's dead-code-by-design (a forward-compat slot) or it's a forgotten plan item. If the former, document it. If the latter, remove it.
- **No test for `MERGE_REPLACE_FLAG` behavior in production paths** (M-1). It's the fallback for unknown Folder Redirection values, and the only way a malformed/empty `display_value` would land here. An integration test against a real folder-redirection GPO with an unknown value would pin it.
- **The `samples/` skip-on-CI scenario dropped coverage from 86.71% to 86.55%** — a 0.16pp gap. The 85% floor has 1.55pp of headroom. If a future PR adds code that depends on samples for coverage, the CI scenario will fall below the floor silently. Consider adding a CI-only assertion that uses synthetic fixture data approximating the samples shape.
- **Two commits ahead of origin before this session** (`b32705f`, `a7f3fcc`) were from prior sessions that didn't push. I pushed everything in one `git push` at the end. If those commits had CI failures, I shipped broken code without seeing the failure. A session-end `git push` is the right gate but assumes the prior commit's CI already passed.
- **`/end` skill's Step 1b `reconcile` is conservative** — it found "no open work items appear resolved in git history" because my commit message didn't say "resolve WI-NNN" with that exact phrase. The reconcile pass is brittle; the manual `close` command was the load-bearing step. Worth understanding whether reconcile can be tuned.
