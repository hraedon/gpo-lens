## On the project

The project is in strong shape for production. The architecture is well-defended: AST-enforced import boundary, defusedxml everywhere, zip-bomb streaming guards, SDDL size caps, DB 0600, CSRF, security headers, audit logging. The plan-driven development pattern (Plans 007–021) with wi_*.md specs for every module is paying dividends — when the glm reviewer traced merge.py's 982 lines, the spec made it navigable.

The two architecture-debt breadcrumbs (web/app.py 1486 lines, queries.py split) are now half-resolved: queries.py was split into a clean package this session, and web/app.py remains open as a medium-priority breadcrumb. Neither was urgent — the code is stable and tested — but the queries split already feels like an improvement in navigability.

The most production-relevant gap is WI-037 (cross-process ingest lock). The current `threading.Lock` is fine for single-worker IIS deployment (which is the documented deployment model), but it's a landmine for anyone who scales to multiple workers. The fix is small (SQLite exclusive transaction), but it touches the ingest hot path and deserves its own focused session.

## On the work done

The three-agent parallel review pattern was effective. Each agent covered a different surface area without overlap, and the findings were complementary: the kimi CI/deploy agent caught infrastructure issues (XML escaping, $dom variable, .gitignore gaps), the glm core agent found logic bugs (CREATE-after-DELETE, float order, depth clamp), and the kimi web agent found security gaps (CSRF method coverage, LLM endpoint validation, HEC URL validation).

The two-round adversarial review (adversarial-reviewer + adversarial-reviewer-2) caught real issues the implementation agents missed:
- `merge_metadata` was the 7th parser missing the non-dict guard — the glm agent fixed 6 but missed this one. The adversarial reviewer caught it by systematically checking every `for record in data:` pattern.
- `HecSink.from_env()` → CLI crash regression — the kimi agent added URL validation in `__init__` but didn't trace the call path through `from_env()` to the CLI. The adversarial reviewer traced it and found the unhandled `ValueError`.
- `json.loads("null")` returning `None` → `tuple(None)` crash — a subtle edge case that neither implementation agent considered. The adversarial reviewer thought through valid-but-wrong-type JSON values.

The Export-Principals.ps1 `$dom` bug was more interesting than the initial review suggested. The kimi agent flagged it as "uninitialized in SddlSource path," but reading the code revealed it was actually a variable reuse bug — `$dom` was used for both the `Get-ADDomain` object (line 70) and the per-SID domain string (line 94), causing the domain object to be clobbered in *both* paths, not just SddlSource. The fix was renaming the per-SID variable to `$domPart` and guarding the domain object access.

I'm confident in all the core module fixes — they're small, targeted, and well-tested. The LLM delimiter hardening (random `<q-{hex}>` tags) is the one I'd want a second pair of eyes on: it's a defense-in-depth improvement, not a bug fix, and the test coverage verifies the mechanism but not the real-world prompt injection resistance.

## Gaps to flag

- **`_get_rw_conn` in web/app.py accesses `wrapped._total` (private attr)** at `app.py:170` — the `SizeLimitedReader._total` attribute is accessed from outside the class. This was noted in the prior reflection and remains. A `total_read` property would be cleaner.
- **Ingest audit log is written before DB commit** at `app.py:1160-1163` — `_events.append_event` runs before `save_estate` returns. If `save_estate` raises after the audit log, the log claims success for a snapshot that wasn't persisted. The audit should move after `save_estate` succeeds.
- **Baseline upload total bytes check is post-write** at `app.py:170-172` — `total_bytes_read` is incremented after `wrapped._total` is read, and the limit check happens after the write completes. An oversized upload's last chunk is already on disk before the exception fires.
- **`narration.py` LLM response parsing assumes Anthropic's exact shape** at `narration.py:86-91` — `content[0].text` indexing will fail if the endpoint returns an error object or a different provider's shape. No validation of `body` shape before indexing.
- **`sinks.py` HEC URL with `http://` sends token in plaintext** at `sinks.py:99` — the `Authorization: Splunk {token}` header is sent over HTTP when `verify_tls=False` or the URL is `http://`. Should warn or reject non-HTTPS HEC URLs when TLS verification is enabled.
- **`Export-GpoEstate.ps1` has empty `catch { }` blocks** at lines 56, 64, 81 — intentional resilience for the collector, but masks systemic issues. Verbose logging inside catch blocks would help operators diagnose repeated failures.
- **`web/app.py` `gpo_detail` doesn't catch `ValueError` from `load_estate`** at `app.py:753` — an empty/corrupt DB yields 500 instead of a user-friendly error. `home` and `danger_list` already catch `ValueError`; `gpo_detail` should too.
- **The `error.html` Jinja2 template bug (`}` instead of `}}`)** was present since the template was created and wasn't caught by any test. Error page rendering for 404/413/500 was broken. Fixed this session, but the fact that it wasn't tested means error pages should have at least one integration test.
